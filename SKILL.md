# 深度研究报告编写方法论

本 skill 将研究报告编写拆解为 **9 个顺序阶段**，前置产物未完成不得进入下一阶段。
核心理念：**先打地基，再盖房子**——资料没抽完不写大纲，事实没核验不写正文，
架构图没画不分章写作。

## 流程总览

```
阶段1: 项目初始化 → 阶段2: 资料抽取与索引 → 阶段3: 事实核验
→ 阶段4: 详细大纲 → 阶段5: 专题研究(卡片+证据包)
→ 阶段6: 核心架构图(先于写作) → 阶段7: 分章写作+数据图表(随写作产出)
→ 阶段8: 红队审查 → 阶段9: 定稿整合
```

每个阶段结束时有**质量门槛检查**，不通过不得进入下一阶段。

> **图表分两批出**：核心架构图（总览图/架构图/流程图）在写作前完成——它们是报告的骨架；数据图表（对比表/趋势图/市场份额图等）随写作按章产出——它们必须与文字内容绑定、用数据驱动，不应提前硬做。

## 阶段 1：项目初始化——理解任务、设定参数

### 1.1 智能推断 + 快速确认

从用户请求中提取已知信息，对未提供参数填入默认值。**一次性展示全部推断结果，只标出需要用户纠正的关键项。**

#### 默认值规则

| 参数 | 默认值 | 覆盖条件 |
|------|--------|---------|
| 题名 | 从请求中提取关键词生成 | 用户明确指定标题 |
| 定位与受众 | 行业决策者 / 投资人 | 用户提到"学术""政府""内部"等信号词时替换 |
| 核心问题 | 题名的疑问句形式（如"XX 的现状、竞争格局与趋势是什么"） | 用户明确提出问题 |
| 规模 | 30–50 页（标准深度） | 用户说"简报""快速"→15页；"深度""全面"→60页+ |
| 交付期限 | 无紧迫期限 | 用户给出具体日期 |
| 已有素材 | 无，完全从公开资料收集 | 用户提供文件/链接 |
| 领域特殊要求 | 无特殊要求 | 用户说明禁忌/偏好/框架 |
| 分析框架 | 自动匹配 2 个最相关维度（见 1.3） | 用户指定框架 |
| 研究方法 | 自动匹配 1–2 个最相关方法（见 1.3 附表） | 主题需要德尔菲法/专家访谈法等依赖用户资源的方法时 |

#### 展示格式

收到用户请求后，用下表一次性展示推断结果：

```
📋 参数确认（直接回复"继续"用默认值启动，或指出需修改的项）

| 参数 | 推断值 | |
|------|--------|---|
| 题名 | 《XX 深度分析报告》 | ✅ |
| 受众 | 行业决策者 | ✅ |
| 研究方法 | 文献研究法 + 比较研究法 | ✅ |
| 核心问题 | XX 的现状、格局与趋势是什么？ | ✅ |
| 规模 | 30–50 页 | ✅ |
| 期限 | 无紧迫期限 | ⚠️ 如有期限请告知 |
| 素材 | 从公开资料收集 | ⚠️ 如有内部资料请提供 |
```

> **规则**：默认 7 项中 ≤ 2 项标 ⚠️。标 ⚠️ 的项是"不说也能继续但说了更好"——不是阻塞条件。

#### 快速启动

用户回复"继续""go""开始"或任何等同信号 → 立即以默认参数进入阶段 1.2（建立工作目录）。只有在用户主动纠正某个参数时才调整。

#### 研究范围确认——仅当用户提供了素材时触发

**触发条件**：用户在发起请求时随附了文件/文件夹/URL 列表（即"已有素材"不为默认值"无"）。未提供素材时跳过本步骤，直接走 1.1 常规确认流程进入阶段 2（此时阶段 2.0 天然会执行完整搜集）。

**为什么必须单独停下来问**：用户提供素材，不代表用户希望"仅基于这些素材写作"——素材可能只是用户手头恰好有的参考资料，报告仍需扩展搜集来补全证据链和独立信源（尤其影响阶段 3 事实核验：如果全部来源都来自用户单方提供，核验台账找不到独立信源交叉验证）。这是一个决定报告可信度基础和工作量的范围决策，不能被默认值或"继续"一并带过。

🛑 **STOP**：检测到素材后，必须先用 AskUserQuestion 单独确认研究范围，不接受笼统的"继续/go/开始"越过此项：

```
📎 检测到您提供了 N 份素材。请明确研究范围：

A. 仅基于您提供的素材（针对性分析报告）
   → 不额外搜集外部资料，核验和分析范围限定在素材内
B. 素材 + 扩展搜集（综合研究报告，推荐）
   → 以您的素材为基础，按阶段 2.0 优先级补充外部来源，
     用于交叉验证和填补素材未覆盖的部分
```

用户选 A → 阶段 2.0 标记为"跳过扩展搜集"，直接进入 2.2 文本抽取，但阶段 3 事实核验时对无法交叉验证的主张标注"仅单一来源（用户提供），未做外部核验"。
用户选 B（或未明确偏好、只说"帮我参考一下"这类模糊表述）→ 阶段 2.0 正常执行完整搜集流程，用户素材与外部搜集结果合并登记到来源索引。

本确认与 1.1 参数确认可以合并展示（同一轮对话内），但**必须等到用户明确二选一后才能进入阶段 1.2**——不因用户只回复"继续"而默认选择 B。

### 1.3 确认分析框架与研究方法

根据报告主题，与用户确认适用的分析框架。通用框架包括：

- **闭环机制**：主题领域的"输入 → 处理 → 输出 → 反馈"闭环是什么？
- **架构分层**：主题可以从哪些架构视角拆解？（如：业务架构 / 技术架构 / 数据架构 / 组织架构 / 治理架构）
- **需求/变化分层**：该领域的变化速度可以分几层？（如：快速响应 / 中期演进 / 长期升级）
- **风险维度**：该领域的主要风险类型有哪些？

> **领域适配**：主题明确时自动匹配 2 个最相关维度（见领域适配指南），不需要逐项询问。主题模糊时默认选"架构分层 + 闭环机制"。

**研究方法**：分析框架回答"怎么切内容"，研究方法回答"怎么收集证据、组织论证"。两者互相独立，同一报告可以任意组合。按主题特点自动匹配 1–2 个方法，写入 1.1 确认表：

| 方法 | 适用场景 | 可行性 | 默认触发条件 |
|---|---|---|---|
| 文献研究法 | 所有报告的基线方法 | 高——web-search-skill + paper-search 直接支撑 | 始终包含 |
| 比较研究法 | 竞争格局/跨国/跨企业/跨方案对比类报告 | 高——多源检索天然支持横向对比 | 主题含"对比""格局""竞争"等信号词 |
| 案例研究法 | 机制拆解、单/多案例深挖类报告 | 高——对应阶段 5 案例卡片，天然契合 | 主题含"案例""机制""怎么做到的"等信号词 |
| 内容分析法 | 政策文本/法规条款/舆情类报告 | 高——MinerU 精准解析 + 结构化抽取支撑 | 主题含"政策""法规""条款"等信号词 |
| 情景分析法 | 趋势预测、战略前瞻类报告 | 中——AI 可基于证据推演情景，但情景假设合理性需人工把关 | 主题含"预测""展望""情景""2030"等未来性信号词 |
| 德尔菲法 / 专家访谈法 | 需要一手专家判断的报告 | 低——依赖用户提供真实专家资源，本 skill 工具链无法独立完成 | 用户明确要求，或主题信息高度稀缺、公开资料无法支撑结论 |

> **默认组合**：无特殊信号词时，默认"文献研究法 + 比较研究法"。命中德尔菲法/专家访谈法触发条件时，在 1.1 确认表对应行标 ⚠️，并向用户说明"需要您提供可联系的专家渠道，否则将降级为文献研究法"。

🔴 CHECKPOINT · STOP：用户回复"继续/开始/go"或纠正完参数后，立即进入阶段 2。即使个别 ⚠️ 项未回复也不阻塞——在后续阶段发现信息不足时再补充。

| **P1 - 应收尽收** | 学术论文（arXiv/PubMed/SSRN 等） | paper-search `search` + `download` | L2 级来源，提供理论和方法论支撑 |

| **P2 - 按需补充** | 深度多源综合研究 | web-search-skill `research` | 当主题跨多个领域、需要综合大量来源时使用 |

#### 工具 1：web-search-skill——通用网页搜索

**脚本路径**：`C:\Users\张\.claude\skills\web-search-skill\scripts\search.js`

**5 个子命令**，按场景选用：

| 子命令 | 用途 | 报告编写中的典型场景 | 命令示例 |
|--------|------|-------------------|---------|
| **search** | 关键词搜索 | 查最新政策、行业数据、新闻报道 | `node …/search.js search "中国商业航天 政策 2025" --max 10` |
| **extract** | 提取已知 URL 的正文内容 | 批量读取已找到的文章/报告原文 | `node …/search.js extract <url1> <url2> --format markdown` |
| **crawl** | 整站爬取 | 爬取某机构网站全部页面（如国务院政策文件库） | `node …/search.js crawl <url> --limit 20 --select-paths /policies/.*` |
| **map** | 列出站点 URL 结构 | 先看网站有哪些页面，再决定爬取哪些 | `node …/search.js map <url>` |
| **research** | 深度多源研究报告（异步，15-60s） | 跨领域综合问题，需要引用标注的报告 | `node …/search.js research "问题" --model pro --length long` |

**常用参数**：
- `--engine auto`（默认）：含中文自动走百度，纯英文走 Tavily
- `--engine both`：中英文同时搜索，两边结果对比（跨国对比话题用）
- `--max N`：返回结果数，默认 10
- `--days N`：限定 N 天内发布的内容（时效性话题用）
- `--json`：输出 JSON，方便程序化处理

**搜索策略**：
1. 先用 **search** 跑 3-5 个核心关键词，广度覆盖
2. 对重要结果用 **extract** 读全文，提取关键数据和判断
3. 仅在以下情况用 **research**：（a）主题跨 ≥3 个领域 或（b）需要深度综合大量来源 → `--model pro --length long`
4. **不要用 research 做简单搜索**——它异步且昂贵，普通查资料走 search

#### 工具 2：paper-search——学术论文搜索与下载

**安装状态**：已安装。CLI 命令为 `paper-search`（`uv tool install paper-search-mcp`），skill 位于 `C:\Users\张\.claude\skills\paper-search\`。

**3 个子命令**，覆盖论文全流程：

| 子命令 | 用途 | 典型命令 |
|--------|------|---------|
| **search** | 多源并发搜索（20+ 平台，自动去重） | `paper-search search "<关键词>" -s arxiv,semantic -n 10` |
| **download** | 下载论文 PDF | `paper-search download arxiv <paper_id> -o research/sources/` |
| **read** | 提取论文全文文本 | `paper-search read arxiv <paper_id>` |

**常用参数**：
- `-s` 指定来源，逗号分隔：`arxiv,semantic,pubmed,crossref,biorxiv,core,zenodo` 等。默认 `all`
- `-n` 每源结果数，默认 5。高效搜索用 `-s arxiv,semantic -n 10`
- `-y` 年份过滤（仅 semantic 源）
- `paper-search sources` 列出全部可用数据源

**使用流程**：
1. 用 `search` 搜论文（优先 `-s arxiv,semantic` 提速，来源更多时加 `crossref,core`）
2. 将返回结果（含标题/作者/年份/摘要/PDF链接/DOI）登记到 `source-index.csv`
3. 如需阅读全文，用 `read` 提取文本；如需存档 PDF，用 `download` 下载到 `research/sources/`
4. 所有 API key 均为可选（Semantic Scholar/CORE/Unpaywall 邮箱等），配置后提升请求限额；不配置也能正常使用

**适用场景**：
- 报告需要学术文献支撑（如技术评估报告的方法论部分）
- 前沿技术领域的文献综述
- 需要引用论文中的实验数据或理论框架

> **与 web-search-skill 的分工**：web-search-skill 用于搜新闻/政策/行业报告/公司信息（以时效性和广度为主），paper-search 用于搜学术论文（以深度和可引用性为主）。不要用一个替代另一个——两者互补。

### 2.1 资料可信度分级（通用标准）

| 等级 | 来源类型 | 用法 |
|---|---|---|
| **A** | 官方文件、法律/法规、原始论文/标准、专利、合同公告、审计报告 | 可作为关键事实依据 |
| **B** | Reuters/AP/FT/Economist 等高公信力媒体、权威行业期刊 | 可作为事件和产业动态依据 |
| **C** | 智库报告、行业分析、公司博客、技术社区、行业协会 | 趋势判断可用，关键事实需交叉验证 |
| **D** | 自媒体、未署名转述、社交媒体、二手编译、逐字稿 | **仅作线索**，不直接入正文 |

> **领域适配**：用户可为特定领域调整 A/B 级来源列表（如医疗领域的 FDA/WHO/NEJM，金融领域的 SEC/央行/审计报告）。

### 2.3 文本抽取——使用 MinerU 精准解析

对搜集到的所有 PDF/DOCX/PPT/HTML/图片素材，使用 MinerU VLM API 进行高精度文本抽取。

**工具**：MinerU 精准解析（`C:\Users\张\.claude\skills\mineru\scripts\mineru_parse.py`）  
**前提**：`MINERU_TOKEN` 环境变量已配置（当前已设置 ✅）  
**能力**：OCR 扫描件识别、公式提取、表格还原、结构化 Markdown 输出

#### 使用方式

```bash
python "C:\Users\张\.claude\skills\mineru\scripts\mineru_parse.py" \
  research/sources/<文件> \
  --output-dir research/extracted \
  --language ch
```

#### 按文件类型选择参数

| 素材类型 | 推荐参数 | 说明 |
|---------|---------|------|
| **PDF（文字型）** | `--language ch` | 中文为主用 `ch`，英文用 `en` |
| **PDF（扫描件/图片型）** | `--language ch`（OCR 默认开启） | 不要加 `--no-ocr`，保持 OCR 开启 |
| **Office 文档**（.docx/.doc/.ppt/.xlsx） | `--language ch` | MinerU 原生支持 Office 格式 |
| **HTML 网页存档** | `--language ch --model-version MinerU-HTML` | HTML 须用专用模型，不支持 `--extra-formats` |
| **图片**（.png/.jpg/.bmp 等） | `--language ch` | 纯图片默认走 VLM OCR |

```
└── <文件名>/
    ├── full.md           # ← 主要使用此文件：结构化 Markdown（含表格/公式/图片）
    ├── images/           #   抽取的图片
    ├── layout.json       #   版面信息
    └── full.docx         #   （可选）--extra-formats docx
```

**抽取后**：将每个文件的 `full.md` 作为后续核验和分析的原文依据。原始素材**不移动、不改名、不删除**。

> **为什么用 MinerU 而不是简单的 `extract`？** web-search-skill 的 `extract` 适合网页正文快速抓取，但 MinerU 对 PDF 扫描件 OCR、表格还原、公式识别的精度远超普通提取——研究报告频繁处理学术论文 PDF 和政府公文扫描件，这些用普通提取会丢失表格数据和公式内容。

### ▶ 阶段 2 质量门槛

- [ ] 资料搜集完成（研究范围为 B 或未提供素材时，已执行 2.0 搜集流程；范围为 A 时已在 1.1a 明确记录）
- [ ] 所有素材完成文本抽取（PDF/Office 类走 MinerU，网页类走 web-search-skill `extract`）
- [ ] 每份素材有完整的元数据记录
- [ ] 每份素材标注了可信度等级（A/B/C/D）
- [ ] D 级材料中的事实陈述已标记为"待核验"
- [ ] 论文类资料（如有）已通过 paper-search 搜索并登记（需 PDF 时用 `download` 下载）

## 阶段 3：事实核验

这是整个方法论的核心——**未经验证的事实不能进入正文**。

### 3.1 建立事实核验台账

创建 `research/claims/claims-ledger.csv`，每条事实主张记录：

| 字段 | 说明 |
|---|---|
| `claim_id` | 唯一编号 |
| `claim_text` | 原文主张 |
| `source_file` | 来源文件名 |
| `claim_type` | 日期 / 数字 / 人物 / 机构 / 事件 / 技术能力 / 因果判断 / 首次性判断 / 其他 |
| `verification_status` | 真实 / 基本属实 / 误导 / 错误 / 无法证实 / 仅为观点 |
| `evidence_sources` | 核验所用的 A/B 级来源 |
| `confidence` | 高 / 中 / 低 |
| `rewrite_suggestion` | 修正后的表述建议 |
| `risk_note` | 如误用会有什么风险 |

### 3.2 核验优先级

优先核验以下类型的主张：
1. **首次性判断**（"首次""第一个""最大"）——必须有 A/B 级独立来源确认
2. **数字与统计**（金额、数量、比例、日期）——必须至少一个 A/B 级来源
3. **因果判断**（"A 导致 B""A 决定了 B"）——必须有 A/B 级来源支持因果链
4. **能力边界**（"完全自主""全自动""秒级"）——几乎总是需要降级为"据称/部分实现"
5. **战果/伤亡/影响**（涉及人的判断）——必须多源交叉验证

### 3.3 核验后处理

| 核验状态 | 正文处理方式 |
|---|---|
| **真实** | 可直接引用，标注来源 |
| **基本属实** | 可引用，加限定词（"据 X 报道""根据 Y 数据"） |
| **误导** | 不入正文，可在附录说明为何误导 |
| **错误** | 不入正文 |
| **无法证实** | 如确需提及，必须显式标注"据称，尚未独立证实" |
| **仅为观点** | 可作为"XX 认为"引用，不能作为事实陈述 |

### ▶ 阶段 3 质量门槛

- [ ] 所有 C/D 级来源中的事实主张已进入台账
- [ ] 台账中"首次/最大/全自动/秒级"等强表述全部完成核验或降级
- [ ] 每个核心章节至少有 3 个 A/B 级来源（可在后续阶段补充）
- [ ] "错误/误导"状态的主张已标记为不入正文

🔴 CHECKPOINT · 🛑 STOP：以上 4 项质量门槛全部通过后进入阶段 4。任一项未通过 → 回到对应子步骤补充（缺来源 → 阶段 2.2 补充资料、缺核验 → 阶段 3.2 优先核验强表述）。

## 阶段 4：详细大纲

结合阶段 1.3 确定的研究方法与分析框架，在事实核验和资料索引的基础上，全面分析报告主题、特色、目的所需要研究和表达的内容，形成**三级标题 + 篇幅建议 + 证据源 + 图表规划**的详细大纲。

### 4.1 大纲结构

```markdown
## 第 X 章：章标题
**本章结论**：一句话（写作时填入）
**篇幅建议**：约 Y 页

### X.1 节标题
**核心论点**：一句话
**关键证据**：来源 ID + 核验状态
**图表规划**：架构图 X-1：描述 / 数据图方向：对比表 1-2 张
**篇幅建议**：约 Z 页

#### X.1.1 小节标题
**核心论点**：一句话
**关键证据**：来源 ID + 核验状态
**篇幅建议**：约 Z 页
```

> **页数不写进标题**：章/节/小节标题本身只写标题文字，不要写成"第 X 章：章标题（建议页数：Y 页）"——大纲标题会原样沿用到成稿目录和正文标题，混入页数会导致最终报告的章节标题里出现"建议 XX 页"这种大纲阶段的内部标注。页数预算统一放在标题下方的"篇幅建议"行，写作阶段参考后即可丢弃，不进入正文。

> **图表规划分两类**：核心架构图（总览图/架构图/流程图）在此阶段就可以给出具体图号和图名——因为阶段 1.3 已确定分析框架，知道有哪些层次和组件需要可视化；**数据图表（对比表/趋势图/雷达图等）只写方向**（如"预计本章需要 1 张市场规模对比表，数据来源 NSR/Euroconsult"），具体图号在阶段 7 随写作产出时才分配——此时写作还没开始，数据细节未定，过早锁定图名会导致图表与文字脱节。

### 4.2 报告结构模板——必选骨架 + 可选模块池

固定模板对所有报告一概适用并不合理：情景前瞻类报告不需要"案例分析"；政策法规类报告需要"条文解读"而非泛化的"背景/现状"；未采用比较研究法的报告不应被迫塞入对比性章节。因此结构不再是单一固定表格，而是**必选骨架决定下限，可选模块按阶段 1.3 确认的研究方法/分析框架/报告目的触发插入**。

#### 必选骨架（任何报告都包含）

| 章节 | 建议占比 | 说明 |
|---|---|---|
| 摘要 | 3–5% | 核心结论 + 关键图表 + 主要启示 |
| 前言/导论 | 5–8% | 问题提出、概念界定、研究边界、方法论（写明 1.3 确定的研究方法） |
| 核心分析 | 35–50% | 报告的技术/业务核心，章数 = 阶段 1.3 分析框架的维度数（通常 2–3 章），每章有架构图 |
| 启示/建议 | 8–12% | 可操作建议，落到组织/流程/技术/人才 |
| 附录 | 5%+ | 资料来源、术语表、图表索引、核验台账摘要 |

#### 可选模块池（按 1.3 的研究方法/分析框架/报告目的触发，插入到骨架之间）

| 模块 | 触发条件（来自 1.3 确认结果） | 建议占比 | 插入位置 |
|---|---|---|---|
| 背景/现状 | 默认包含——纯情景前瞻/条文解读类报告可压缩并入导论 | 10–15% | 导论之后 |
| 案例分析 | 研究方法含"案例研究法" | 15–20% | 核心分析之前 |
| 政策/条文解读 | 研究方法含"内容分析法"，或主题属于政策法规类 | 10–15% | 背景之后、核心分析之前 |
| 情景推演 | 研究方法含"情景分析法" | 10–15% | 核心分析之后 |
| 风险/挑战 | 分析框架含"风险维度"，或报告目的偏决策支持 | 5–10% | 核心分析之后 |
| 横向对比 | 研究方法含"比较研究法"且对比是报告主要目的 | 融入核心分析，或独立成章 10–15% | 核心分析内，或作为独立章 |

> 占比合计不强制等于 100%——各模块占比为参考区间，按主题实际权重裁量分配，未触发的模块不占篇幅。多个可选模块同时触发时（如案例研究法 + 比较研究法同时命中），优先合并为同一批章节（如"案例横向对比"），不机械叠加两套模块。

> **与旧模板的区别**：旧模板把"案例分析""风险挑战"当成放之四海而皆准的必选项；新设计只有 5 项真正通用，其余模块的取舍直接绑定 1.3 已确认的研究方法和分析框架——选了案例研究法，大纲自然有案例分析章；没选情景分析法，就不硬凑"情景展望"一章。

### 4.3 图表规划——分两批，不同精度

阶段 4 列图表清单，但精度不同：

| 图表类型 | 规划精度 | 说明 |
|---------|---------|------|
| **核心架构图**（总览/架构/流程） | 图号 + 图名 + 核心要素 | 基于阶段 1.3 分析框架，此时已明确。例：`图 2-1：六层技术架构全景（感知层/网络层/平台层/数据层/算法层/应用层）` |
| **数据图表**（对比/趋势/份额/雷达） | 方向 + 数据来源 | 写作未开始，数据细节未定。例：`方向：1 张市场区域对比表 + 1 张增长率折线图，数据来源 NSR 2025` |

**数据图表的图号在阶段 7 出图时分配**，不在阶段 4 锁定。阶段 4 只确保"不临时加图"——每章需要什么类型的数据可视化、数据从哪来，心里有数即可。

### ▶ 阶段 4 质量门槛

- [ ] 章节结构（必选骨架 + 可选模块）已依据 1.3 的研究方法、分析框架、报告目的选定，非固定套用模板
- [ ] 大纲含三级标题（章→节→小节）+ 篇幅建议 + 证据源标注
- [ ] 核心架构图清单完整（每张有图号/图名/核心要素）
- [ ] 数据图表方向清单完整（每章标注需要什么类型的图 + 数据来源）
- [ ] 每章有明确的"本章结论"占位
- [ ] 用户确认了大纲结构

🔴 CHECKPOINT · 🛑 STOP：用户确认大纲结构（含章标题/篇幅建议/证据源/图表规划）后进入阶段 5。未确认 → 回到阶段 4.1 调整大纲结构或补充证据源。

## 阶段 5：专题研究——卡片与证据包

卡片是**事实核验台账（阶段 3）和正文写作（阶段 7）之间的中间产物**：台账里是零散的、逐条的事实主张，写作时需要的是"一个案例/一项技术/一套架构/一个概念"的完整叙事单元。卡片把台账中相关的主张组织成可直接转写为正文的结构化单元，同时不丢失每条判断背后的证据链。

### 5.1 卡片类型与下游用途

每类卡片对应下游一个明确的使用环节，不是产出后自由存放：

| 卡片类型 | 内容 | 下游用途 |
|---|---|---|
| **案例卡** | 名称、时间线、问题、动作、技术支撑、效果、证据包、风险争议、可提炼机制 | 直接填充 4.2 可选模块中的"案例分析"章节（命中案例研究法时）；每张卡对应案例分析章节的一节 |
| **技术/主题卡** | 核心概念、架构位置、输入输出、关键数据、证据包、与其他组件关系、局限 | 支撑核心分析章节的机制阐述——写作时逐条转写"输入输出/关键数据/局限"为正文的机制描述 |
| **架构卡** | 架构名称、所属层次、组件与关系、数据流、证据包、验证方式、对应的图表 | **阶段 6 出图的直接依据**——"组件与关系""数据流"是绘制架构图的素材来源，不是阶段 4.3 图表清单能替代的细节 |
| **理论/概念卡** | 概念定义、提出者/来源、证据包、应用场景、与其他概念关系、争议 | 支撑前言/导论的概念界定部分，及核心分析中对理论框架的引用 |

### 5.2 证据包——卡片与事实核验台账的绑定方式

**证据包**：卡片中每条关键判断对应的最小引用单元 = `claim_id`（指向 `claims-ledger.csv` 的核验记录）+ 该记录的 `evidence_sources`（指向 `source-index.csv` 的来源）。

不用笼统的"证据等级：中"这样一个印象分描述整张卡片——卡片里通常有 3-5 条不同强度的判断（例如案例卡的"效果"是强判断需要 A/B 级证据，"风险与争议"可能只有 C 级来源），**每条关键判断单独标注证据包**，卡片整体的可信度是这些证据包的最低值，不是平均值或主观估计。

### 5.3 卡片模板（四类，均含证据包字段）

案例卡：
```markdown
## 案例名称
1. **时间线**：
2. **背景问题**：证据包 [claim_id]
3. **关键动作/适变**：证据包 [claim_id]
4. **技术/业务支撑**：证据包 [claim_id]
5. **效果（量化优先）**：证据包 [claim_id]
6. **风险与争议**：证据包 [claim_id]
7. **可提炼机制**：
```

技术/主题卡：
```markdown
## 技术/主题名称
1. **核心概念**：
2. **架构位置**：（在阶段 1.3 分析框架中所属的层次）
3. **输入输出**：证据包 [claim_id]
4. **关键数据**：证据包 [claim_id]
5. **与其他组件关系**：
6. **局限**：证据包 [claim_id]
```

架构卡：
```markdown
## 架构名称
1. **所属层次**：（对应阶段 1.3 分析框架的哪一层）
2. **组件与关系**：（用于阶段 6 出图的核心素材）
3. **数据流**：证据包 [claim_id]
4. **验证方式**：（该架构判断如何被核实，如官方文档/逆向分析/访谈）
5. **对应的图表**：（图号，阶段 6 出图时回填）
```

理论/概念卡：
```markdown
## 概念名称
1. **定义**：证据包 [claim_id]
2. **提出者/来源**：
3. **应用场景**：
4. **与其他概念关系**：
5. **争议**：证据包 [claim_id]
```

### 5.4 卡片索引登记——让"卡片有没有被用上"可追溯

每张卡片产出后，登记到 `research/notes/card-index.csv`（模板见 [`references/card-index-template.csv`](references/card-index-template.csv)），字段：

`card_id, card_type, title, chapter_ref, evidence_packet_refs, credibility_summary, status, used_in_chapter`

- `card_id`：卡片唯一编号（如 CASE-01、ARCH-02）
- `chapter_ref`：对应大纲中的章节号（写卡片时填，来自阶段 4 大纲）
- `evidence_packet_refs`：该卡片关联的 claim_id 列表（分号分隔）
- `status`：draft / final
- `used_in_chapter`：**阶段 7 写作时回填**——写完一章后，将该章实际引用过的卡片 ID 标记对应章节号；未使用则留空

**先登记再写作**。卡片不登记直接拿去写，等于放弃了阶段 8 检查"哪些调研成果被浪费"的能力。

### ▶ 阶段 5 质量门槛

- [ ] 每个核心章节至少有 3 张专题卡片
- [ ] 每张卡片的关键判断均标注了证据包（claim_id），不是笼统的单一证据等级
- [ ] 卡片之间无矛盾（如有矛盾已在卡片中标注）
- [ ] 所有卡片已登记到 `card-index.csv`，`chapter_ref` 字段完整

## 阶段 6：核心架构图——先于写作

**核心架构图必须在分章写作之前完成**。它们是报告的骨架——定义分析框架、拆解层次、展示逻辑链路，文字要围绕它们展开。

### 6.1 出图清单

从阶段 4.3 的"核心架构图"清单取出图号和图名，再从阶段 5 的**架构卡**（`research/notes/architecture-cards/`）取出"组件与关系""数据流"作为具体绘制素材——阶段 4.3 只锁定"要画什么图"，架构卡才是"图里具体画什么"的依据。按此优先级：

1. **总览图**（1–3 张）—— 全文核心判断、主题闭环图，读者看完就理解报告主线
2. **架构图**（3–6 张）—— 主题的 X 层架构拆解，报告的分析框架
3. **流程图**（3–5 张）—— 关键流程的端到端展示

### 6.2 出图工具——选择与使用

根据图的类型选择合适的工具：

| 图类型 | 首选工具 | 输出格式 | 保存路径 |
|--------|---------|---------|---------|
| **架构图、流程图**（分层/组件/连接关系复杂） | drawio | `.drawio` + `.svg` | `research/figures/` |
| **技术架构图**（系统拓扑/技术栈分层/部署架构） | fireworks-tech-graph | `.svg` + `.png` | `research/figures/` |
| **简单流程图/时序图**（线性逻辑） | Mermaid（内联到 Markdown） | 无需独立文件 | — |

## 阶段 7：分章写作与数据图表

写作和数据图表**同步推进**——每完成一章的文字草稿，立即产出该章的数据图表。这样图表与文字在同一个上下文中形成，不会出现脱节。

### 7.1 写作标准

1. **证据驱动**：每个关键判断对应至少一个证据包（阶段 5.2 定义：claim_id + 来源），正文引用时格式化为"来源机构/媒体 + 日期 + 标题"
2. **不确定性显式标注**：不把"可能/据称/尚未证实"包装成确定事实
3. **机制导向**：用"机制 / 链路 / 流程 / 约束 / 指标"表达，少用名词堆砌
4. **不写两张皮**：技术章节给业务含义，业务章节有技术支撑
5. **每章结构统一**：开头"本章结论" → 正文 → 末尾"对主论点的贡献"

### 7.2 写作顺序

按大纲顺序逐章推进。**每章的写作循环**：

```text
调取专题卡片 → 写完文字草稿 → 立即产出该章的数据图表 → 图表嵌入文字 → 检查证据源 → 检查台账 → 回填卡片使用记录 → 进入下一章
```

具体步骤：
1. **调取该章对应的专题卡片**：按 `card-index.csv` 的 `chapter_ref` 找到本章卡片，卡片是本章的一手素材——案例卡的"时间线/动作/效果"直接转写为案例分析段落，技术卡的"输入输出/局限"直接转写为机制阐述，不重新凭记忆组织
2. 完成当前章的**文字草稿**（不含图表）
3. **立即产出该章的数据图表**：写作中遇到对比数据 → 出对比表；遇到时间序列 → 出折线图；遇到占比 → 出饼图
4. 将图表**嵌入**文字中对应位置，确保首次引用在图表之前
5. 检查该章的证据源是否都标注了可信度等级
6. 检查是否出现了阶段 3 台账中标记为"不入正文"的主张
7. 检查首次性/最强级表述是否都有 A/B 级证据
8. **回填 `card-index.csv`**：本章实际引用过的卡片，`used_in_chapter` 填入章节号；卡片内容与本章无关或被弃用，`used_in_chapter` 留空但不删除卡片（供阶段 8 判断是否遗漏）
9. 通过后进入下一章

**为什么用这个顺序**：写作时脑中有数据、有上下文，此时出图最自然——图表不是"补"上去的，而是思维的自然延伸。提前在阶段 6 做的数据图表容易出现"图表是对的但文字用不上"或"文字需要一张图但之前没做"的情况。

### 7.3 报告内容质量标准——不只是"不出错"，还要"写得好"

过程严谨保证下限，内容质量决定上限。以下标准用于每章写作后的自查，也用于阶段 8 红队审查时的内容维度打分。

#### 标准 A：证据密度

**每个自然段至少对应 1 个来源引用。** 不允许出现连续 3 段以上无来源的"流畅叙述"。

自查方法：逐段检查，在段落末尾标注来源编号。如果发现连续 3 段无法标注任何来源——这些段落很可能是写作者自己的推测，需要补充证据或删除。

#### 标准 B：主张—证据—推理链

**每个关键判断必须呈现完整的三角结构：主张 → 证据 → 推理。**

❌ 只有主张无证据：
> 东南亚新能源汽车市场正处于爆发前夜。

✅ 完整三角：
> **主张**：东南亚新能源汽车市场正进入快速增长期。
> **证据**：据泰国工业联合会 2025 年报，泰国 2025 年 BEV 销量同比增长 240%，充电桩数量从 2024 年的 2,800 个增至 2025 年的 9,000+ 个。
> **推理**：销量 3 倍增长 + 基础设施同步扩张 → 不只是政策补贴的短期效应，而是供需双轮驱动的结构性增长。

#### 标准 C：建议的可操作性

**启示/建议章节的每一条必须回答：谁来做？做什么？在什么条件下？用什么资源？预期效果如何衡量？**

❌ 空泛建议：
> 企业应加强本地化布局。

✅ 可操作建议：
> **谁**：海外事业部战略规划组。
> **做什么**：2026 Q4 前完成泰国 Rayong 的 KD 工厂选址（优先考察 WHA 工业区，因比亚迪/长城已入驻形成配套集群）。
> **资源**：初期投入约 2-3 亿泰铢（~600 万美元），需董事会批准海外投资额度。
> **效果指标**：工厂投产后 12 个月内本地化率 > 40%，享受泰国 EV 3.5 计划全额税收优惠。

#### 标准 D：量化优先

**能用数字就不用形容词。** 优先用数量、比例、时间、金额、排名来表达。

| 弱表述 | 强表述 |
|---|---|
| "大幅增长" | "同比增长 240%" |
| "显著缩短" | "从 30 分钟缩短至 2 分钟" |
| "成本较高" | "TCO 比方案 B 高约 35%（5 年期）" |
| "很多企业已布局" | "截至 2026Q1，至少 8 家中国 Tier-1 零部件企业在泰国设厂" |

#### 标准 E：承认边界

**每章的"对主论点的贡献"段落中，必须包含至少 1 句对本章分析局限的说明。**

示例：
> 本章分析基于截至 2026 年 5 月的公开资料。三地试点政策仍在快速演变中（成都于 2026 年 4 月发布了新版空域管理细则，因截稿时间未能纳入详细分析）。此外，由于无法获得三地政府内部决策文件，制度逻辑的分析主要基于公开政策文本和二手访谈，存在推断成分。

#### 标准 F：摘要的可独立阅读性

**摘要不是目录的散文版，而是报告的浓缩版。** 读者只读摘要也应该能理解：背景、问题、方法、核心发现（3-7 条）、主要建议、关键约束。

每个核心发现用 **加粗一句话** 开头，后面跟 2-3 句支撑。

### 7.4 写作禁区

- 未完成事实核验的主张直接写入正文
- 把 D 级来源当作结论使用
- 把领域核心概念写成泛泛的科普（必须落到机制层面）
- 使用"完全""首次""秒级""决定性"等强表述而无 A/B 级证据
- 只堆材料不做机制提炼，或只谈概念不给案例/数据/图表
- 写"据外媒报道""有消息称"等空泛来源——必须写具体媒体+日期+标题

### ▶ 阶段 7 质量门槛

- [ ] 每章开头有"本章结论"，末尾有"对主论点的贡献"（含至少 1 句局限说明）
- [ ] 所有 C/D 级来源的事实已标注强度（据称/尚未证实）
- [ ] 所有图表已在正文中引用（图在首次引用的段落之后）
- [ ] 每章数据图表与文字内容一致（数字/名称/单位不矛盾）
- [ ] 本章对应的专题卡片已全部核对使用情况（引用的已回填 `used_in_chapter`，未引用的有合理原因）
- [ ] 未出现台账中标记为"错误/误导"的主张
- [ ] **证据密度检查**：抽查 3 段连续文字，均能找到对应来源
- [ ] **建议可操作性检查**：每条建议回答了谁/做什么/资源/指标四个问题

## 常见问题

**Q：用户只给了一个主题，没有提供素材怎么办？**
A：按阶段 2.0「资料搜集」执行——先用 web-search-skill `search` 搜 A/B 级来源（官方文件/权威媒体），再用 `paper-search` CLI 搜学术论文。详细优先级和命令示例见阶段 2.0。

**Q：用户提供了素材，是不是就不用扩展搜集了？**
A：不是默认跳过。阶段 1.1a 会强制停下来问用户："仅基于素材（针对性分析）"还是"素材+扩展搜集（综合研究，推荐）"——用户提供素材只说明"有这些"，不代表"只要这些"。未明确选择前不进入阶段 2。

**Q：用户要求的时间很紧，能否跳过某些阶段？**
A：可以压缩但不能跳过。最精简路径：阶段 3（事实核验）和阶段 8（红队审查）不可跳过——前者防止传播错误信息，后者防止论证塌方。阶段 5（专题卡片）可合并到阶段 7（写作）中。

**极速模式**（时间 < 3 天或页数 < 20 页）：
- 阶段 1：自动填入默认参数，只确认题名后直接启动
- 阶段 2-3：合并为"资料收集+交叉验证"，边收集边核验，用 claims-ledger 只记录高风险主张
- 阶段 4：大纲降到二级标题，不设页数预算
- 阶段 5：跳过专题卡片，直接在阶段 7 写作中做案例嵌入
- 阶段 6：只出 1 张总览图，架构图和流程图用文字描述替代
- 阶段 7：写作与数据图表同步，写作标准 A-F 中只强制执行 A（证据密度）和 D（量化优先）
- 阶段 8：红队审查压缩为 3 项（证据薄弱/夸大表述/矛盾）
- 阶段 9：跳过 Word 导出，交付 Markdown 终稿

> 极速模式不是偷懒——是承认约束、做有意识的取舍。但阶段 3（事实核验）和阶段 8（红队审查）仍不可跳过。

**Q：报告不需要架构图怎么办？**
A：如果主题确实不需要架构图（如纯叙事/历史类报告），可以在阶段 1 确认。但大多数分析型报告至少需要 1 张核心闭环图，帮助读者在 30 秒内理解报告主线。

**Q：用户中途改变了想法（主题/规模/受众）？**
A：回到阶段 1 重新确认参数，检查哪些已有产物可以复用，哪些需要重做。不在新旧要求之间折中妥协。

## 反例清单——执行本 skill 时不要做的事

| # | 反模式 | 为什么不能做 | 正确做法 |
|---|--------|------------|---------|
| 1 | **参数确认阻塞流程** | 逐项列出 7 个问题让用户逐一回答，首轮交互零产出——用户说"帮我写报告"，你回"请先回答 7 个问题" | 阶段 1.1 自动推断全部参数，一次性展示，≤2 项标 ⚠️。用户说"继续"即启动 |
| 2 | **跳过参数直接搜资料** | 没确认受众/核心问题就搜，搜回来的素材大概率偏题或粒度不对 | 至少确认题名和受众后再进阶段 2。默认值也是确认——它是主动选择，不是偷懒 |
| 3 | **D 级来源当结论用** | 自媒体/社交媒体的断言未经核验，写入正文会传播错误信息 | D 级来源仅作线索，必须找到 A/B 级来源确认后才能入正文 |
| 4 | **质量门槛未通过就进入下一阶段** | "先写着再说"会导致整篇报告建立在未核验事实上，后期返工成本更高 | 见上方「质量门槛失败处理」表，按症状→处理→退回阶段执行 |
| 5 | **先写正文再补图表** | 写完所有文字后再回头加图表，文字和图表脱节——图是对的但文字用不上，或者文字需要一个图但没做 | 核心架构图在阶段 6 先于写作完成；数据图表在阶段 7 随写作按章产出（写完一章→立即做该章的数据图表） |
| 6 | **把强表述当成事实** | "首次""最大""完全自主"等措辞在法律和商业语境下有法律责任和误导风险 | 核验台账中专门盯首次性判断/能力边界类主张，找不到 A/B 级证据就降级 |
| 7 | **红队审查走形式** | 作者自己审查自己的报告有严重盲区，容易放过逻辑跳跃和夸大表述 | 至少做一轮独立红队（换视角/假设对立立场），高风险项必须全部处理 |
| 8 | **只写正向不写边界** | 报告读起来像宣传稿而非研究报告，损害可信度 | 每章末尾强制留至少 1 句分析局限说明（写作标准 E） |
| 9 | **建议写成空话** | "加强/优化/提升"没有主语/动作/资源/指标的虚假可操作性 | 每条建议必须回答：谁 / 做什么 / 资源 / 效果指标（写作标准 C） |
| 10 | **跳过 Word 导出** | 最终交付物格式不规范降低专业信任度 | 阶段 9 必须执行 `scripts/markdown_to_docx.py`，生成符合 V3.0 规范的 .docx |
