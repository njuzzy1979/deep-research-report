# 架构分析方法论指南

## 什么是架构分析

架构分析不是画一张"系统组成图"，而是**从多个视角拆解一个复杂系统的结构、关系、流动和约束**。
好的架构分析让读者理解：系统由哪些部分组成、它们如何协作、变化发生在哪里、瓶颈在哪里。

---

## 通用架构分析视角

根据报告主题，从以下视角中选择 3–6 个（不必全部使用）：

### 1. 业务/领域架构
- **问题**：业务/领域的核心流程是什么？从输入到输出经过哪些环节？
- **要素**：角色、活动、决策点、信息流、价值流
- **产出**：端到端流程图、决策树、价值链图

### 2. 技术/系统架构
- **问题**：支撑业务的技术系统如何分层？组件之间如何通信？
- **要素**：层次（基础设施/平台/应用/接口）、组件、协议、依赖
- **产出**：分层架构图、组件交互图

### 3. 数据架构
- **问题**：数据从哪里来、如何流动、在哪里存储、怎样被消费？
- **要素**：数据源、数据流、存储、转换、治理、质量
- **产出**：数据流图、数据模型图

### 4. 组织/治理架构
- **问题**：谁负责什么？决策权如何分配？协调机制是什么？
- **要素**：角色、职责、汇报线、委员会、流程
- **产出**：组织架构图、决策矩阵、RACI 表

### 5. 安全/风险架构
- **问题**：系统面临什么威胁？如何防护？失败后如何恢复？
- **要素**：威胁面、防护层、检测机制、响应流程、恢复策略
- **产出**：威胁模型图、防护层次图

### 6. 演进/迭代架构
- **问题**：系统如何随时间变化？不同变化速度的组件如何共存？
- **要素**：变化触发源、变化速度分层、测试-部署-回滚机制
- **产出**：演进路线图、分层迭代模型

---

## 架构分析的五个标准问题

每个架构视角（或本章核心技术章节）必须回答以下五个问题：

1. **位置**：该组件/子系统在总体架构中的位置是什么？上下游是什么？
2. **输入输出**：该组件接收什么输入？产生什么输出？格式和约束是什么？
3. **依赖**：该组件依赖哪些数据、链路、基础设施、外部系统？
4. **验证**：该组件如何被测试和验证？正确性、性能、安全性各用什么方法？
5. **反馈**：该组件的运行结果如何反馈到系统其他部分？如何触发迭代？

## 架构图绘制指南

### 图的前提

一张好的架构图应该满足：
- **自明性**：读者不看正文也能大致理解
- **层次感**：核心组件突出，辅助组件淡出
- **一致性**：同类型组件用同形状/同色系
- **有限性**：一张图不超过 15–20 个主要节点

### 图的标注

每张架构图应包含：
- 图标题（图 X-Y：描述）
- 图例（颜色/形状/线型的含义）
- 关键注释（约束条件、前提假设、数据来源）

### 图的检查

- [ ] 图中的每个节点在正文中都有对应说明？
- [ ] 图中的每条连线（数据流/依赖）在正文中都有解释？
- [ ] 图与图之间的逻辑一致（同一对象在不同图中名称相同、关系一致）？
- [ ] 图的复杂度与章节篇幅匹配（不要用一张极其复杂的图来"省文字"）？

## 从原方法论中抽象出的通用模式

### 模式 1：闭环反馈

几乎任何领域的深度分析都可以用一个闭环图来贯穿：

```
[触发/变化] → [感知/识别] → [分析/决策] → [行动/执行] → [效果评估] → [知识沉淀] → 回到[触发/变化]
```

根据主题调整各环节的名称和具体内容。

### 模式 2：分层迭代

大多数复杂系统都有不同速度的变化层：

| 层级 | 变化速度 | 典型动作 | 验证强度 | 部署策略 |
|---|---|---|---|---|
| 快速响应 | 小时–天 | 参数调整、规则更新 | 轻量校验 | 全局快速生效，可回滚 |
| 中期演进 | 天–周 | 组件升级、模型微调 | 自动化+对抗测试 | 先试点后推广 |
| 长期升级 | 周–月 | 架构变更、新系统集成 | 全场景验证 | 分级灰度 |

> 根据主题调整：企业产品可以是"热修复/功能迭代/架构升级"，公共政策可以是"紧急令/行政令/立法"。

### 模式 3：多视角架构拆解

不要只从一个角度描述系统。至少从 3 个视角交叉分析，每个视角有专属图表。

## 不同领域的架构适配示例

### 企业/产业分析
- 业务架构：客户旅程 → 价值交付流程
- 技术架构：IT 系统分层 → 数据中台 → 业务中台
- 竞争架构：市场格局 → 差异化定位 → 护城河
- 财务架构：收入模型 → 成本结构 → 现金流

### 公共政策分析
- 法律架构：上位法 → 行政法规 → 地方细则
- 行政架构：中央 → 省市 → 基层的权责与资源
- 利益相关方架构：政府 / 企业 / 公众 / 媒体的博弈关系
- 执行架构：政策工具 → 试点 → 评估 → 推广

### 技术/产品评估
- 功能架构：核心功能 → 扩展功能 → 接口
- 技术架构：前端 → 后端 → 数据 → 基础设施
- 安全架构：身份认证 → 授权 → 审计 → 加密
- 迭代架构：需求管理 → 开发流程 → 测试 → 部署 → 监控
