## 1.1 智能推断 + 快速确认

从用户请求中提取已知信息，对未提供参数填入默认值。**一次性展示全部推断结果，只标出需要用户纠正的关键项。**

### 报告类型自动识别

从用户请求中识别报告类型，自动匹配模板结构。不同类型的报告有不同的结构需求——立项报告需要技术指标量化表和创新点归纳，政策分析需要条文解读框架，技术评估需要评测方法论。

| 报告类型 | 信号词 | 默认模板 | 特殊模块 |
|---------|--------|---------|---------|
| **立项报告/论证报告** | "立项"、"论证"、"项目建议书"、"可行性"、"申报"、"申请书" | 必选骨架 + 立项专用可选模块集 | 技术指标与考核方式、创新点归纳、TRL评估、进度里程碑、研究基础 |
| **深度研究报告**（默认） | "深度分析"、"研究报告"、"白皮书"、"行业分析" | 当前必选骨架 + 当前可选模块池 | 无特殊新增 |
| **政策分析报告** | "政策"、"法规"、"条文"、"治理" | 必选骨架 + 政策专用可选模块集 | 条文解读方法、利益相关方矩阵、政策影响评估框架 |
| **技术评估报告** | "技术评估"、"选型"、"对比"、"评测" | 必选骨架 + 技术专用可选模块集 | 评测方法论、基准测试设计 |
| **快速简报** | "简报"、"快速"、"概览"、"brief" | 极速模式压缩模板 | 无特殊模块——自动触发极速模式 |

若用户未使用任何信号词，默认归类为"深度研究报告"。

### 默认值规则

| 参数 | 默认值 | 覆盖条件 |
|------|--------|---------|
| 题名 | 从请求中提取关键词生成 | 用户明确指定标题 |
| 定位与受众 | 行业决策者 / 投资人 | 用户提到"学术""政府""内部"等信号词时替换 |
| 核心问题 | 题名的疑问句形式（如"XX 的现状、竞争格局与趋势是什么"） | 用户明确提出问题 |
| 规模 | 30–50 页（标准深度） | 用户说"简报""快速"→15页；"深度""全面"→60页+ |
| 交付期限 | 无紧迫期限 | 用户给出具体日期 |
| 已有素材 | 无，完全从公开资料收集 | 用户提供文件/链接 |
| 领域特殊要求 | 无特殊要求 | 用户说明禁忌/偏好/框架 |
| 分析框架 | 自动匹配 2 个最相关维度（见 1.3） | 用户指定框架 |
| 研究方法 | 自动匹配 1–2 个最相关方法（见 1.3 附表） | 主题需要德尔菲法/专家访谈法等依赖用户资源的方法时 |

### 展示格式

收到用户请求后，用下表一次性展示推断结果：

```
📋 参数确认（直接回复"继续"用默认值启动，或指出需修改的项）

| 参数 | 推断值 | |
|------|--------|---|
| 题名 | 《XX 深度分析报告》 | ✅ |
| 受众 | 行业决策者 | ✅ |
| 研究方法 | 文献研究法 + 比较研究法 | ✅ |
| 核心问题 | XX 的现状、格局与趋势是什么？ | ✅ |
| 规模 | 30–50 页 | ✅ |
| 期限 | 无紧迫期限 | ⚠️ 如有期限请告知 |
| 素材 | 从公开资料收集 | ⚠️ 如有内部资料请提供 |
```

> **规则**：默认 7 项中 ≤ 2 项标 ⚠️。标 ⚠️ 的项是"不说也能继续但说了更好"——不是阻塞条件。

### 快速启动

用户回复"继续""go""开始"或任何等同信号 → 立即以默认参数进入阶段 1.2（建立工作目录）。只有在用户主动纠正某个参数时才调整。

### 研究范围确认——仅当用户提供了素材时触发

**触发条件**：用户在发起请求时随附了文件/文件夹/URL 列表（即"已有素材"不为默认值"无"）。未提供素材时跳过本步骤，直接走 1.1 常规确认流程进入阶段 2（此时阶段 2.0 天然会执行完整搜集）。

**为什么必须单独停下来问**：用户提供素材，不代表用户希望"仅基于这些素材写作"——素材可能只是用户手头恰好有的参考资料，报告仍需扩展搜集来补全证据链和独立信源（尤其影响阶段 3 事实核验：如果全部来源都来自用户单方提供，核验台账找不到独立信源交叉验证）。这是一个决定报告可信度基础和工作量的范围决策，不能被默认值或"继续"一并带过。

🛑 **STOP**：检测到素材后，必须先用 AskUserQuestion 单独确认研究范围，不接受笼统的"继续/go/开始"越过此项：

```
📎 检测到您提供了 N 份素材。请明确研究范围：

A. 仅基于您提供的素材（针对性分析报告）
   → 不额外搜集外部资料，核验和分析范围限定在素材内
B. 素材 + 扩展搜集（综合研究报告，推荐）
   → 以您的素材为基础，按阶段 2.0 优先级补充外部来源，
     用于交叉验证和填补素材未覆盖的部分
```

用户选 A → 阶段 2.0 标记为"跳过扩展搜集"，直接进入 2.2 文本抽取，但阶段 3 事实核验时对无法交叉验证的主张标注"仅单一来源（用户提供），未做外部核验"。
用户选 B（或未明确偏好、只说"帮我参考一下"这类模糊表述）→ 阶段 2.0 正常执行完整搜集流程，用户素材与外部搜集结果合并登记到来源索引。

本确认与 1.1 参数确认可以合并展示（同一轮对话内），但**必须等到用户明确二选一后才能进入阶段 1.2**——不因用户只回复"继续"而默认选择 B。

## 1.2 建立工作目录与工具路径配置

在项目根目录创建 `research/` 目录骨架：

```text
research/
  sources/          # 来源元数据索引（不移动原始文件）
  extracted/        # 抽取的文本
  claims/           # 事实核验台账
  notes/
    case-cards/     # 案例卡片
    tech-cards/     # 技术/主题卡片
    architecture-cards/  # 架构卡片
    theory-cards/   # 理论/概念卡片
    card-index.csv  # 卡片索引：登记每张卡片的类型/对应章节/关联证据包/是否已被阶段7引用
  figures/          # 图表（草稿 + 终稿）
  drafts/           # 分章草稿
```

**工具路径配置**：外部工具的脚本路径集中存放在 [`references/tool-paths.json`](tool-paths.json) 中。首次使用时，用户需将其中 `absolute_path` 字段填写为实际的绝对路径（安装路径），或留空表示该工具未安装/暂不使用。

配置方式（任选其一）：
- **手动编辑**：直接修改 `references/tool-paths.json` 中的 `absolute_path` 字段
- **自动检测**：skill 执行时尝试 `where`/`which` 查找工具路径，将结果写入 `tool-paths.json`

配置完成后，阶段 2/6 中调用外部工具的命令将自动读取 `tool-paths.json` 中的路径，无需在 SKILL.md 中硬编码。

> **示例**：如果 MinerU 安装在 `D:\tools\mineru\scripts\mineru_parse.py`，则将 `"absolute_path": ""` 改为 `"absolute_path": "D:\\\\tools\\\\mineru\\\\scripts\\\\mineru_parse.py"`（注意 JSON 中转义反斜杠）。

### 封面生成

参数确认后，自动生成 `research/cover.md`，格式为纯 YAML frontmatter。

将 1.1 中确认的 title（题名）、subtitle（英文副标题，如有）、report_type（报告类型）、org（机构名）、date（日期）、version（版本号）、header_short（页眉简称）等字段写入封面文件：

```markdown
---
title: 《XX 深度分析报告》
title_en: English Title (if applicable)
report_type: 深度研究报告
org: 遨天科技
date: 2026年7月
version: V1.0
header_short: 页眉简称
---
```

生成后展示给用户确认（"请确认封面信息，后续修改只需编辑 `research/cover.md`"）：

```
📋 封面信息确认

| 字段 | 值 | |
|------|----|---|
| title | 《XX 深度分析报告》 | ✅ |
| title_en | English Title | ⚠️ 如有英文标题请告知 |
| report_type | 深度研究报告 | ✅ |
| org | 遨天科技 | ⚠️ 如有申报单位请告知 |
| date | 2026年7月 | ✅ |
| version | V1.0 | ✅ |
| header_short | 页眉简称 | ⚠️ 用于页眉显示 |
```

用户回复"确认"→ 保存 `research/cover.md`。用户修改某个字段 → 更新后重新确认。

**封面文件独立于正文 Markdown**：后续阶段如需修改封面信息（如标题调整、版本号更新），只需编辑 `research/cover.md` 这一文件，不影响正文草稿。转换器通过 `--cover research/cover.md` 参数读取封面字段，优先级高于正文 YAML frontmatter。封面未生成的默认字段沿用 1.1 确认表中的推断值，用户不确认不阻塞后续阶段。

## 1.3 确认分析框架与研究方法

根据报告主题，与用户确认适用的分析框架。通用框架包括：

- **闭环机制**：主题领域的"输入 → 处理 → 输出 → 反馈"闭环是什么？
- **架构分层**：主题可以从哪些架构视角拆解？（如：业务架构 / 技术架构 / 数据架构 / 组织架构 / 治理架构）
- **需求/变化分层**：该领域的变化速度可以分几层？（如：快速响应 / 中期演进 / 长期升级）
- **风险维度**：该领域的主要风险类型有哪些？

> **领域适配**：主题明确时自动匹配 2 个最相关维度（见领域适配指南），不需要逐项询问。主题模糊时默认选"架构分层 + 闭环机制"。

**研究方法**：分析框架回答"怎么切内容"，研究方法回答"怎么收集证据、组织论证"。两者互相独立，同一报告可以任意组合。按主题特点自动匹配 1–2 个方法，写入 1.1 确认表：

| 方法 | 适用场景 | 可行性 | 默认触发条件 |
|---|---|---|---|
| 文献研究法 | 所有报告的基线方法 | 高——web-search-skill + paper-search 直接支撑 | 始终包含 |
| 比较研究法 | 竞争格局/跨国/跨企业/跨方案对比类报告 | 高——多源检索天然支持横向对比 | 主题含"对比""格局""竞争"等信号词 |
| 案例研究法 | 机制拆解、单/多案例深挖类报告 | 高——对应阶段 5 案例卡片，天然契合 | 主题含"案例""机制""怎么做到的"等信号词 |
| 内容分析法 | 政策文本/法规条款/舆情类报告 | 高——MinerU 精准解析 + 结构化抽取支撑 | 主题含"政策""法规""条款"等信号词 |
| 情景分析法 | 趋势预测、战略前瞻类报告 | 中——AI 可基于证据推演情景，但情景假设合理性需人工把关 | 主题含"预测""展望""情景""2030"等未来性信号词 |
| 德尔菲法 / 专家访谈法 | 需要一手专家判断的报告 | 低——依赖用户提供真实专家资源，本 skill 工具链无法独立完成 | 用户明确要求，或主题信息高度稀缺、公开资料无法支撑结论 |

> **默认组合**：无特殊信号词时，默认"文献研究法 + 比较研究法"。命中德尔菲法/专家访谈法触发条件时，在 1.1 确认表对应行标 ⚠️，并向用户说明"需要您提供可联系的专家渠道，否则将降级为文献研究法"。

🔴 CHECKPOINT · STOP：用户回复"继续/开始/go"或纠正完参数后，立即进入阶段 2。即使个别 ⚠️ 项未回复也不阻塞——在后续阶段发现信息不足时再补充。
