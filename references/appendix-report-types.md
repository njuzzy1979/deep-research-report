## A. 报告类型自动识别（阶段 1.1）

从用户请求中识别报告类型，自动匹配模板结构。不同类型的报告有不同的结构需求。

| 报告类型 | 信号词 | 默认模板 | 特殊模块 |
|---------|--------|---------|---------|
| **立项报告/论证报告** | "立项"、"论证"、"项目建议书"、"可行性"、"申报"、"申请书" | 必选骨架 + 立项专用可选模块集 | 技术指标与考核方式、创新点归纳、TRL评估、进度里程碑、研究基础 |
| **深度研究报告**（默认） | "深度分析"、"研究报告"、"白皮书"、"行业分析" | 当前必选骨架 + 当前可选模块池 | 无特殊新增 |
| **政策分析报告** | "政策"、"法规"、"条文"、"治理" | 必选骨架 + 政策专用可选模块集 | 条文解读方法、利益相关方矩阵、政策影响评估框架 |
| **技术评估报告** | "技术评估"、"选型"、"对比"、"评测" | 必选骨架 + 技术专用可选模块集 | 评测方法论、基准测试设计 |
| **快速简报** | "简报"、"快速"、"概览"、"brief" | 极速模式压缩模板 | 无特殊模块——自动触发极速模式 |

若用户未使用任何信号词，默认归类为"深度研究报告"。

## B. 报告类型→模板映射表（阶段 4.2）

阶段 1.1 识别报告类型后，按以下映射自动加载对应的模板结构。`struct_template` 是阶段 4 大纲产出的内部标识，决定了可选模块池中哪些模块默认触发、哪些模块被禁用。

| 报告类型 | struct_template | 默认触发的可选模块 | 禁用的可选模块 | 特殊写作要求 |
|---------|----------------|-------------------|---------------|-------------|
| **立项报告** | `proposal` | 背景/现状 + 风险/挑战 | — | 技术指标必须量化（数值+单位+达标判定标准）；TRL评估必须引用标准定义；创新点必须独立成节并区分"理论创新/技术创新/应用创新" |
| **深度研究报告**（默认） | `research` | 背景/现状 | — | 需要架构图（至少1张）、证据驱动写作、机制导向分析 |
| **政策分析** | `policy` | 政策/条文解读 + 背景/现状（压缩入导论） | 案例分析、情景推演 | 条文解读必须注明条款编号和出处；利益相关方分析必须包含"受益方/受损方/中立方"三分法 |
| **技术评估** | `tech-eval` | 背景/现状 + 风险/挑战 | — | 评测方法论必须声明比较基准和度量指标；基准测试结果必须注明测试环境和条件 |
| **快速简报** | `brief` | 无（全部压缩） | 全部可选模块 | 自动触发极速模式（阶段 2-3 合并、大纲降到二级标题、只出 1 张总览图） |

### 立项报告专用可选模块（5 个新增）

当 `struct_template = proposal` 时，以下模块自动追加到可选模块池：

| 模块 | 建议占比 | 插入位置 | 核心内容要求 |
|------|---------|---------|-------------|
| 技术指标与考核方式 | 5–8% | 研究内容之后 | 每项指标含：指标名称、目标值、考核方法、达标判定标准；技术就绪度不同的指标分期考核 |
| 创新点归纳 | 3–5% | 技术途径之后 | 分"理论创新/技术创新/应用创新"三类；每项创新点含：创新内容、与现有技术/方法的具体区别、预期效益 |
| TRL成熟度评估 | 3–5% | 技术途径之后 | 按国军标或NASA TRL定义逐项评估当前成熟度和目标成熟度；附TRL提升路径和关键验证节点 |
| 进度计划与里程碑 | 3–5% | 核心分析之后 | 含甘特图或里程碑表；每阶段有明确的交付物和完成标志；标注关键路径和里程碑之间的依赖关系 |
| 研究基础与条件 | 3–5% | 进度计划之后 | 已有研究成果、实验条件、团队能力、合作单位；说明与本研究的关系和支撑作用 |

> **模块触发优先级**：`struct_template` 默认触发 > 1.3 研究方法触发 > 用户在大纲确认时手动调整。

## C. 领域适配指南

当用户的研究主题属于特定领域时，按以下规则直接替换框架参数。领域明确时自动应用，领域不明确时在阶段 1 确认。

### 适配 1：替换可信度 A/B 级来源列表

在阶段 1 确认领域后，将对应来源列表写入 `source-index.csv`：

| 领域 | A 级（官方/权威）示例 | B 级（高公信力）示例 |
|------|---------------------|---------------------|
| 医疗/生物 | FDA、WHO、NEJM、Lancet | NIH、CDC、PubMed 索引期刊 |
| 金融/经济 | SEC、央行、审计报告、上市公司年报 | Bloomberg、FT、WSJ |
| 军事/国防 | 国防白皮书、DOD 官方公告、国际条约 | Jane's、RUSI、CSIS |
| 科技/IT | 原始论文（arXiv/ACM/IEEE）、专利、标准（ISO/W3C） | MIT Tech Review、IEEE Spectrum |

> 用户未提供领域特定列表时，使用阶段 2.1 的通用标准（A=官方文件/法规/论文/专利，B=高公信力媒体/权威期刊）。

### 适配 2：替换架构分析维度

根据主题从下方映射表选择，替换阶段 1.3 的通用架构维度：
- **企业/产业分析**：商业模式架构 / 技术架构 / 数据架构 / 组织架构 / 竞争架构
- **公共政策**：法律架构 / 行政架构 / 财政架构 / 社会影响架构 / 执行架构
- **技术评估**：系统架构 / 数据架构 / 安全架构 / 部署架构 / 演进架构

### 适配 3：替换需求/变化分层

根据主题从下方映射表选择变化分层：
- **企业战略**：战术调整（周）/ 战略调整（月）/ 转型（年）
- **技术产品**：热修复（小时-天）/ 功能迭代（周-月）/ 架构升级（月-季）
- **公共政策**：紧急措施（天）/ 行政令（周-月）/ 立法（月-年）

### 适配 4：替换闭环模型

根据主题从下方映射表选择核心闭环：
- **军事**：感知 → 认知 → 决策 → 执行 → 反馈
- **企业**：市场信号 → 战略调整 → 产品研发 → 上市验证 → 效果反馈
- **技术**：需求 → 设计 → 开发 → 测试 → 部署 → 监控 → 迭代
- **政策**：问题识别 → 方案设计 → 试点 → 评估 → 推广/调整
