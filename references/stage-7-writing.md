写作和数据图表**同步推进**——每完成一章的文字草稿，立即产出该章的数据图表。这样图表与文字在同一个上下文中形成，不会出现脱节。

## 7.0 卡片到正文的叙事化转写

卡片是研究笔记，正文是论证性写作。转写的核心原则：

**1. 以论点为主线，不以卡片为主线**
- 先确定本节要论证什么（来自大纲的"本节要论证什么"）
- 从卡片中选取支撑论点的素材
- 一张卡片的不同字段可以分散在本节的不同位置
- 不要求每张卡片都被正文引用——不是所有调研成果都要入正文

**2. 三条转写铁律**
- **禁止将卡片的字段逐条翻译为段落**：案例卡的"时间线/背景问题/关键动作/技术支撑/效果/风险争议/可提炼机制"是分析框架，不是正文的7个自然段
- **禁止在正文中出现字段标签**：正文中不应出现"时间线：2022年..."、"效果（量化优先）：增长240%"这类卡片模板痕迹
- **每条关键判断必须呈现完整的三角结构**：主张→证据→推理。卡片提供了"证据"那一角，写作时补全"主张"和"推理"

**3. 卡片到段落的正确转写路径**

| 卡片字段组合 | 正文转写产物 | 示例 |
|------------|------------|------|
| 时间线 + 背景问题 + 关键动作 | 案例叙述段落（按时间或因果展开） | "2022年SpaceX首次演示...随后在2023年..." |
| 核心概念 + 输入输出 + 关键数据 | 技术机制阐述段落 | "TLE轨道数据通过SGP4模型传播后..." |
| 可提炼机制 / 关键数据 | 观点/结论段落（作为论证的"证据"角） | "这表明AI云探测已在时效性上超越传统方法..." |
| 风险争议 + 局限 | 边界/不确定性说明段落 | "然而该方案在低轨目标上的精度仍受限于..." |

**4. used_in_chapter 回填的强制执行**
- 写完一章后，必须逐张检查本章对应的卡片（通过 card-index.csv 的 `chapter_ref` 字段）
- **引用了的卡片**：在 `used_in_chapter` 填入章节号（如"第3章"）
- **未引用的卡片**：在 `card-index.csv` 的 `notes` 列注明原因（如"素材重复""与本章论点无关""证据不足弃用"）
- 阶段7质量门槛第5项强制执行此检查

> **反例警示**：如果正文中出现了"时间线：2023年X月..."、"关键动作：部署..."这样的字段标签式写作，说明写作者是在"翻译卡片"而非"用卡片论证"——退回到本节重新理解"以论点为主线"原则。

## 7.1 写作标准

1. **证据驱动**：每个关键判断对应至少一个证据包（阶段 5.2 定义：claim_id + 来源），正文引用时格式化为"来源机构/媒体 + 日期 + 标题"
2. **不确定性显式标注**：不把"可能/据称/尚未证实"包装成确定事实
3. **机制导向**：用"机制 / 链路 / 流程 / 约束 / 指标"表达，少用名词堆砌
4. **不写两张皮**：技术章节给业务含义，业务章节有技术支撑
5. **每章结构统一**：开头"本章结论" → 正文 → 末尾"对主论点的贡献"

## 7.2 编号使用规范——哪些地方要编号，哪些不要

编号系统的技术实现（图表编号格式、章节自动编号、脚注、参考文献）已由 [`references/研究报告格式规范.md`](研究报告格式规范.md) 第四节统一规定，写作时不需要重新设计格式。本节规范的是**正文内容组织层面**的编号取舍——同一批材料，拆成编号列表还是写成连贯段落，直接决定条理性和可读性。

### 应当使用编号的场景

| 场景 | 编号方式 | 说明 |
|---|---|---|
| 机制/流程拆解（≥3 个有先后或依赖关系的步骤） | 阿拉伯数字 1. 2. 3. 或①②③ | 呼应标准 3"机制导向"：如"①边缘节点压缩回传→②知识图谱映射→③模型生成推荐→④人工确认下发"，编号体现步骤间的先后/依赖关系 |
| 启示/建议章节的每条建议 | "**建议 N：标题**" | 呼应标准 9，编号让读者能在结论/摘要中精确回指某一条建议 |
| 摘要的核心发现 | "**核心发现 N**：" | 呼应标准 12，3–7 条为宜，每条独立编号便于单独引用 |
| 红队风险清单条目 | R001、R002…（阶段 8 已定义） | 沿用 `red-team-checklist.md` 既有格式，不重复设计 |
| 图 / 表 / 公式 | 图 X-Y / 表 X-Y / 式(X-Y)（章号-序号，格式规范 4.2） | 写作阶段图号只在阶段 7 出图时分配，草稿中可先用占位描述，定稿前统一核对连续性 |

### 不应手动编号的场景

| 场景 | 原因 | 正确做法 |
|---|---|---|
| 章 / 节 / 小节标题 | 阶段 9 转换脚本按 `references/研究报告格式规范.md` 自动生成"第一章 / 1.1 / 1.1.1"；写作时若手动加编号（如标题写成"1. 导论"），会与自动编号叠加产生"第一章 第一章：导论"这类重复（详见 `references/md-to-docx-pitfalls.md` 问题 4） | 标题只写文字本身，不加任何数字/中文数字前缀，编号完全交给阶段 9 转换脚本 |
| 并列且无顺序/权重差异的要点 | 编号隐含"有先后"或"有优先级"，材料本身没有这层关系时会误导读者 | 用无序列表（`-` / `•`），不用阿拉伯数字 |
| 主张—证据—推理的论证性段落 | 标准 8 要求呈现完整推理链，拆成编号列表会把因果论证切碎成孤立要点，读者看不到"为什么" | 保持连贯段落叙述，不为了"显得有条理"而机械编号化 |

### 正文引用编号的规范

- 引用图表一律用具体编号（"如图 3-2 所示"），不用"下图""上表"这类依赖排版位置的说法——Word 排版分页后图表实际位置会变化，位置性指代会失效
- 引用建议/风险/发现编号时全文必须一致（如结论或附录中回指"建议 3"，需与该建议在原章节中的编号一致），阶段 9 整合时一并核对

## 7.2.2 分章文件中的标题书写规则（与 md→docx 转换器的合约）

**核心原则：分章文件中的标题只写纯文字，不加任何数字前缀或中文数字编号。所有编号（"第一章/1.1/1.1.1"）完全由阶段9转换脚本自动生成。**

| 层级 | Markdown | 正确示例 | 错误示例 | 说明 |
|------|----------|---------|---------|------|
| **前言/导论** | `#` (H1) | `# 前言/导论` | `# 0. 前言` | 全文唯一H1，也是转换器识别的MAIN_TITLE |
| **前言各节** | `##` (H2) | `## 问题提出与研究背景` | `## 0.1 问题提出` | 不加"0.1/0.2"编号，转换器识别为前置件不编号 |
| **正文各章** | `##` (H2) | `## 军事需求分析` | `# 第一章 军事需求分析` 或 `## 1. 军事需求分析` | **正文不使用H1**，全部使用H2且不加任何编号 |
| **正文各节** | `###` (H3) | `### 非合作目标异动意图判断` | `### 1.1 非合作目标异动` | H3不加"1.1/1.2"编号 |
| **正文各小节** | `####` (H4) | `#### 轨道残差异常快速检测` | `#### 1.1.1 轨道残差` | H4不加"1.1.1"编号 |
| **附录** | `##` (H2) | `## 附录A：事实核验台账摘要` | — | H2，转换器识别APPENDIX |

**为什么正文不使用 H1？** 转换器将第一个 H1 视为主标题（MAIN_TITLE），后续 H1 全部降级为 CHAPTER 处理。如果正文各章使用 H1（如`# 第一章 xxx`），将触发 W-HDR-03 警告，且可能导致编号错乱。正文各章统一使用 H2 可以避免这个问题。

**分文件写作时的注意事项**：当报告超过一定规模需要按章拆分为多个 Markdown 文件时，每个文件独立使用 `##`（H2）开头。合并为 `final-report.md` 后，所有 H2 被转换器按顺序正确编号。不要在分章文件中写 `# 第X章` 这样的 H1 标题——合并后会触发多个主线 H1 的冲突。

## 7.2.3 每章完成后的即时自查（不通过不解锁下一章）

每章草稿完成后，必须按以下清单逐项自查并打勾。这是阶段7质量门槛的**前置执行机制**——不依赖阶段7全部完成后才检查，而是在每一章的写作循环末尾即时触发。

- [ ] **本章结论段落**存在且为3-5句话概括（非一句话敷衍）
- [ ] **对主论点的贡献段落**存在且含至少1句局限说明（写作标准E）
- [ ] **证据密度检查**：随机抽查3段连续文字，每段均能标注来源（写作标准A）
- [ ] **强表述检查**：全文搜索"首次/最大/完全/秒级/决定性/全自动"，每条均有A/B级证据（写作禁区第4条）
- [ ] **所有C/D级来源**的事实已标注强度限定词（"据称""尚未证实""据X报道"）（写作禁区第2条）
- [ ] **所有引用**使用具体来源格式（"据X机构 Y年报告《标题》"），无"据外媒报道""有消息称"等空泛来源（写作禁区第6条）
- [ ] **所有图表已在正文中引用**（"如图X-Y所示"），首次引用在图之前（阶段7步骤4）
- [ ] **本章对应的专题卡片已全部核对使用情况**并回填 `card-index.csv` 的 `used_in_chapter` 字段（阶段7步骤8）
- [ ] **编号列表审计**：搜索"① ② ③"和"1. 2. 3."→确认编号列表仅用于机制/流程拆解场景，非无顺序差异的要点堆砌（§7.2）
- [ ] **逻辑词审计**：搜索"因此""显然""必然"→确认每个词前后的逻辑链条完整，无跳跃（阶段8.1"逻辑跳跃"预防）

**以上10项全部通过 → 进入下一章。任一项未通过 → 在当前章修复后重检，不得跳过修复进入下一章。**

> **与阶段7质量门槛的关系**：本自查清单是每章写作循环内的即时检查（逐章触发），阶段7末尾的质量门槛（见下方）是全部分章完成后的汇总检查（一次性触发）。两者互补：自查清单防止"写到最后一章才发现第一章有问题"，质量门槛确保"全报告交付前整体一致"。

## 7.3 写作顺序

按大纲顺序逐章推进。**每章的写作循环**：

```text
调取专题卡片 → 写完文字草稿 → 立即产出该章的数据图表 → 图表嵌入文字 → 检查证据源 → 检查台账 → 回填卡片使用记录 → 进入下一章
```

具体步骤：
1. **调取该章对应的专题卡片**：按 `card-index.csv` 的 `chapter_ref` 找到本章卡片，卡片是本章的一手素材——案例卡的"时间线/动作/效果"直接转写为案例分析段落，技术卡的"输入输出/局限"直接转写为机制阐述，不重新凭记忆组织
2. 完成当前章的**文字草稿**（不含图表）
3. **立即产出该章的数据图表**：
   a. **加载报告图表样式模板**：
      ```python
      import matplotlib.pyplot as plt
      plt.style.use('design/chart-quality-constraints/matplotlib-report-style.mplstyle')
      ```
      此模板设定了全报告统一的字体（宋体+TNR）、字号（标签10pt/刻度9pt/图例9pt）、
      灰度色板（7档）、300dpi 等全局参数。**每张图出图前必须先执行此语句**。
   b. 选择图表类型——对照 `design/chart-quality-constraints/00-chart-quality-design.md`
      第 3.4 节的"图表类型选择决策表"，禁止使用 3D 图表和 >5 扇区的饼图
   c. 写作中遇到对比数据 → 出对比表；遇到时间序列 → 出折线图；遇到占比 → 出饼图
      （≤5 项；>5 项用横向条形图）。所有数据图表用 `dpi=300` + `bbox_inches='tight'`
      保存到 `research/figures/`，文件命名 `<图号>-<描述>.png`
   d. **颜色注册**：若图中出现了新的概念/实体（`color-registry.csv` 中无记录），
      在 `color-registry.csv` 中登记其颜色映射
4. 将图表**嵌入**文字中对应位置，确保首次引用在图表之前
5. 检查该章的证据源是否都标注了可信度等级
6. 检查是否出现了阶段 3 台账中标记为"不入正文"的主张
7. 检查首次性/最强级表述是否都有 A/B 级证据
8. **回填 `card-index.csv`**：本章实际引用过的卡片，`used_in_chapter` 填入章节号；卡片内容与本章无关或被弃用，`used_in_chapter` 留空但不删除卡片（供阶段 8 判断是否遗漏）
9. 通过后进入下一章

**为什么用这个顺序**：写作时脑中有数据、有上下文，此时出图最自然——图表不是"补"上去的，而是思维的自然延伸。提前在阶段 6 做的数据图表容易出现"图表是对的但文字用不上"或"文字需要一张图但之前没做"的情况。

## 7.4 报告内容质量标准——不只是"不出错"，还要"写得好"

过程严谨保证下限，内容质量决定上限。以下标准用于每章写作后的自查，也用于阶段 8 红队审查时的内容维度打分。

### 标准 A：证据密度

**每个自然段至少对应 1 个来源引用。** 不允许出现连续 3 段以上无来源的"流畅叙述"。

自查方法：逐段检查，在段落末尾标注来源编号。如果发现连续 3 段无法标注任何来源——这些段落很可能是写作者自己的推测，需要补充证据或删除。

### 标准 B：主张—证据—推理链

**每个关键判断必须呈现完整的三角结构：主张 → 证据 → 推理。**

❌ 只有主张无证据：
> 东南亚新能源汽车市场正处于爆发前夜。

✅ 完整三角：
> **主张**：东南亚新能源汽车市场正进入快速增长期。
> **证据**：据泰国工业联合会 2025 年报，泰国 2025 年 BEV 销量同比增长 240%，充电桩数量从 2024 年的 2,800 个增至 2025 年的 9,000+ 个。
> **推理**：销量 3 倍增长 + 基础设施同步扩张 → 不只是政策补贴的短期效应，而是供需双轮驱动的结构性增长。

### 标准 C：建议的可操作性

**启示/建议章节的每一条必须回答：谁来做？做什么？在什么条件下？用什么资源？预期效果如何衡量？**

❌ 空泛建议：
> 企业应加强本地化布局。

✅ 可操作建议：
> **谁**：海外事业部战略规划组。
> **做什么**：2026 Q4 前完成泰国 Rayong 的 KD 工厂选址（优先考察 WHA 工业区，因比亚迪/长城已入驻形成配套集群）。
> **资源**：初期投入约 2-3 亿泰铢（~600 万美元），需董事会批准海外投资额度。
> **效果指标**：工厂投产后 12 个月内本地化率 > 40%，享受泰国 EV 3.5 计划全额税收优惠。

### 标准 D：量化优先

**能用数字就不用形容词。** 优先用数量、比例、时间、金额、排名来表达。

| 弱表述 | 强表述 |
|---|---|
| "大幅增长" | "同比增长 240%" |
| "显著缩短" | "从 30 分钟缩短至 2 分钟" |
| "成本较高" | "TCO 比方案 B 高约 35%（5 年期）" |
| "很多企业已布局" | "截至 2026Q1，至少 8 家中国 Tier-1 零部件企业在泰国设厂" |

### 标准 E：承认边界

**每章的"对主论点的贡献"段落中，必须包含至少 1 句对本章分析局限的说明。**

示例：
> 本章分析基于截至 2026 年 5 月的公开资料。三地试点政策仍在快速演变中（成都于 2026 年 4 月发布了新版空域管理细则，因截稿时间未能纳入详细分析）。此外，由于无法获得三地政府内部决策文件，制度逻辑的分析主要基于公开政策文本和二手访谈，存在推断成分。

### 标准 F：摘要的可独立阅读性

**摘要不是目录的散文版，而是报告的浓缩版。** 读者只读摘要也应该能理解：背景、问题、方法、核心发现（3-7 条）、主要建议、关键约束。

每个核心发现用 **加粗一句话** 开头，后面跟 2-3 句支撑。

## 7.5 写作禁区

- 未完成事实核验的主张直接写入正文
- 把 D 级来源当作结论使用
- 把领域核心概念写成泛泛的科普（必须落到机制层面）
- 使用"完全""首次""秒级""决定性"等强表述而无 A/B 级证据
- 只堆材料不做机制提炼，或只谈概念不给案例/数据/图表
- 写"据外媒报道""有消息称"等空泛来源——必须写具体媒体+日期+标题
