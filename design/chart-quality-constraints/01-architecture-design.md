# 架构设计文档：图表绘制质量约束方案

## 1. 架构概述

### 1.1 定位

本约束方案在 deep-research-report skill 的现有架构中位于**质量门禁层**——它不是独立新增的阶段，而是对阶段 6（核心架构图）和阶段 7（数据图表）现有质量门槛的**结构化深化**。当前质量门槛仅覆盖分辨率/文件格式（300dpi pHYs 检查），本方案将其扩展为覆盖配色、字体、一致性、可访问性、图表类型选择的完整约束体系。

在现有 9 阶段架构中的位置：

```
阶段 5 专题研究
    ↓
阶段 6 核心架构图 ──→ [新] 架构图约束（配色/字体/一致性/可访问性）  ← 本方案增强
    ↓
阶段 7 分章写作+数据图表 ──→ [新] 数据图约束（配色/字体/类型选择/可访问性） ← 本方案增强
    ↓
阶段 8 红队审查 ──→ [新] 图表一致性检查（跨图颜色映射注册表验证） ← 本方案增强
    ↓
阶段 9 定稿整合 ──→ [新] 图表合规性最终检查（自动+人工） ← 本方案增强
```

### 1.2 架构模式

**模式**：约束注册表 + 分层映射（Constraint Registry + Layered Adapter）

理由：4 种出图工具的控制力差异极大（matplotlib 可全局 rcParams 控制 → Mermaid 只能通过主题变量），无法用单一配置格式覆盖所有工具。采用"中心注册表 + 工具适配层"模式：
- **中心注册表**定义约束的"意图"（如"背景色=白色 #FFFFFF，主要文本色=黑色 #000000"）
- **工具适配层**将意图翻译为每个工具可执行的配置（matplotlib rcParams、drawio Mermaid 主题变量、fireworks JSON 参数、Mermaid 主题配置）

此模式保证了：
1. 跨工具约束的一致性（同一色值在所有工具中表达相同语义）
2. 新增工具时只需添加适配层，不修改中心注册表
3. 约束可验证——检查脚本只需读取中心注册表 + 工具产物即可判断合规性

### 1.4 核心设计原则

1. **V3.1 格式规范是最高权威**：本方案所有约束的 MUST 级基线直接引用 V3.1 §5.2 现有规定（300dpi、14cm/16cm 宽度、黑白灰主色、≥9pt 文字、10pt 坐标轴标签、9pt 图例），不重复定义也不与之冲突。
2. **渐进约束**：MUST 级阻塞流程，SHOULD 级记录偏差但不阻塞，MAY 级仅为建议。避免过严约束导致工具可用性下降。
3. **中心注册、工具适配**：色值、字体规格只在约束注册表中定义一次，各工具的配置由适配层翻译。
4. **检查自动化优先**：能用脚本验证的约束优先自动化，无法自动化的才降级为人工检查清单项。
5. **向后兼容**：新增约束不改变现有工作流的产物格式和工具调用方式，仅在关键节点插入加载/检查步骤。

## 2. 约束分层模型

约束体系分为 **3 层**，自下而上依次为：

```
┌──────────────────────────────────────────────────┐
│  Layer 3: 验证与执行层 (Enforcement Layer)         │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 自动检查  │ │ 人工清单  │ │ 阶段8红队一致性   │  │
│  │ 脚本     │ │ (Checklist)│ │ 检查             │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
├──────────────────────────────────────────────────┤
│  Layer 2: 规范与适配层 (Specification Layer)       │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ rcParams │ │ drawio    │ │ fireworks JSON   │  │
│  │ 配置模板  │ │ 样式约束  │ │ 样式参数模板     │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │

│  │         Mermaid 主题配置模板              │    │
│  └──────────────────────────────────────────┘    │
├──────────────────────────────────────────────────┤

│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 配色注册表│ │ 字体注册表│ │ 图表类型决策矩阵  │  │

│  │         跨图颜色映射注册表                 │    │
│  │         (cross-figure-color-map)          │    │
│  └──────────────────────────────────────────┘    │
└──────────────────────────────────────────────────┘
```

### Layer 1: 策略与注册表层（Policy & Registry）

**职责**：定义"要约束什么"——是约束体系的单一事实来源（Single Source of Truth）。此层不包含任何工具特定的实现细节。

**内容**：
- **配色注册表**：定义语义色值（`主要文本色`、`次要文本色`、`强调色`、`系列色 1-6`、`背景色`、`网格线色`、`边框色`）、灰度打印等效色值、色盲无障碍色值对、暗/亮打印适配建议
- **字体注册表**：定义各工具的统一字体族、字号规格（标题/标签/图例/注释/坐标轴标签）、回退策略（西文优先 + 中文 fallback 链）
- **图表类型决策矩阵**：数据类型 → 推荐图表类型 → 反模式的二维矩阵，附每类图的代码骨架索引
- **跨图颜色映射注册表**：记录"同一语义实体在不同图中使用相同颜色"的映射关系——如"感知层"在所有架构图中始终为同一颜色

**与其他层的关系**：
- Layer 2 从 Layer 1 **读取**约束值，翻译为工具配置
- Layer 3 从 Layer 1 **读取**期望值，对照工具产物判断合规性
- Layer 1 本身不执行任何操作，仅在约束更新时被修改

### Layer 2: 规范与适配层（Specification & Adapter）

**职责**：将 Layer 1 的"意图约束"翻译为每个工具可执行的"操作配置"。每个工具对应一个适配模块。

**内容**：
- **matplotlib rcParams 模板文件**：完整的 `matplotlibrc` 风格文件，将配色注册表中的色值映射为 `axes.edgecolor`、`grid.color`、`text.color` 等参数，将字体注册表映射为 `font.family`、`font.size` 系列参数
- **drawio 样式约束规格**：Mermaid 主题变量映射（`%%{init: {'theme':'base', 'themeVariables': {...}}}%%`）+ 基础 mxGraphModel 样式片段模板（用于通过 XML 精细控制时注入）
- **fireworks-tech-graph 样式参数模板**：基于 style-1-flat-icon.md 的色值，将语义色值映射为 JSON `nodes[].style` 和 `arrows[].style` 字段的参数片段
- **Mermaid 主题配置模板**：仅用于 ≤15 节点的简单流程图，将配色注册表映射为 `themeVariables` 字典

**与 Layer 1 的关系**：适配层**引用但不复制** Layer 1 的色值。理想情况下，适配层配置由脚本从注册表自动生成，避免手动同步导致偏离。

### Layer 3: 验证与执行层（Enforcement）

**职责**：在流程的特定节点（阶段 6 质量门禁、阶段 7 每章完成、阶段 8 红队审查、阶段 9 最终检查）执行约束合规性验证。

**内容**：
- **自动检查脚本**：检查可量化约束（PNG dpi、图片尺寸、色值是否在注册表允许范围内、字体大小是否 ≥ 最低值、pHYs 元数据存在性）
- **人工检查清单**：检查无法自动化的约束（"图表间同一实体的颜色是否一致"、"图表类型的选用是否与决策矩阵推荐吻合"、"图注是否清晰可读"）
- **阶段 8 红队一致性检查**：在现有 7 个检查维度基础上新增"图表一致性"维度，专门检查跨图颜色映射注册表的一致性

**与 Layer 1/2 的关系**：验证层读取 Layer 1 的期望值 + Layer 2 的工具配置 + 实际产物文件，三向对比得到合规性判断。

## 4. 约束等级体系

### 4.1 等级划分原则

| 等级 | 语义 | 判定标准 | 违规处理 |
|------|------|---------|---------|
| **MUST** | 阻塞性 | (a) 已在 V3.1 格式规范 §5.2 中明确规定；或 (b) 违反后将导致 docx 嵌入失败或打印不可读 | 自动检查 FAIL → **退回当前阶段**，修复后重检 |
| **SHOULD** | 不阻塞但记录偏差 | 显著影响专业性但不导致技术故障（如配色偏离灰度色板、字体不一致） | 自动检查 WARN → **记录到偏差日志**，进入下一阶段，偏差在阶段 8 红队审查时逐条确认 |
| **MAY** | 建议 | 提升图表质量但对核心约束无影响（如色盲无障碍增强、特殊配色主题） | 不做自动检查，在人工检查清单中作为"建议"项出现 |

### 4.2 各维度归属

#### 配色规范

| 约束 | 等级 | 理由 |
|------|------|------|
| 绘图区域背景色 = 白色 (#FFFFFF) | MUST | V3.1 §5.2 "黑白灰为主" |
| 主要文本/线条使用黑色 | MUST | V3.1 §5.2 |
| 多系列数据使用灰度渐变区分（≥ 2 级灰度差异） | MUST | V3.1 "辅助可用深灰色对比强调" |
| 不使用红绿作为唯一的系列区分手段 | SHOULD | 色盲无障碍——不影响印刷可读性但影响可访问性 |
| 彩色强调色仅用于 ≤ 1 个关键数据系列 | SHOULD | 保持印刷一致性 |
| 提供 CUD（色觉障碍友好）的替代配色 | MAY | 可访问性增强 |

#### 字体与字号规范

| 约束 | 等级 | 理由 |
|------|------|------|
| 图内文字 ≥ 9pt | MUST | V3.1 §5.2 明确规定 |
| 坐标轴标签 = 10pt | MUST | V3.1 §5.2 明确规定 |
| 图例 = 9pt | MUST | V3.1 §5.2 明确规定 |
| 图宽 = 14cm 时，文字在缩放后保持可读 | MUST | V3.1 尺寸约束的推论——如果 14cm 宽下 9pt 不可读则约束无效 |
| 中文字体使用黑体系列（SimHei/微软雅黑/PingFang SC） | SHOULD | 与 V3.1 §3 标题体系一致 |
| 西文数字使用 Times New Roman 或等宽字体 | SHOULD | 与 V3.1 §3 字体体系一致 |
| 同一报告中所有图表使用统一的字体族 | SHOULD | 跨图一致性 |

#### 图表间一致性规则

| 约束 | 等级 | 理由 |
|------|------|------|
| 同一语义实体在不同图中使用相同颜色 | SHOULD | 专业性要求，不满足会导致阅读混乱 |
| 同一语义实体在不同图中使用相同名称 | MUST | V3.1 阶段 9.1 "统一术语"的自然延伸 |
| 跨图颜色映射注册表在阶段 6 首次出图前创建 | MUST | 先注册后使用——无注册表就无法验证一致性 |
| 阶段 8 红队审查时逐条验证注册表一致性 | SHOULD | 人工判断为主 |

#### 技术格式约束

| 约束 | 等级 | 理由 |
|------|------|------|
| PNG 300dpi+，含 pHYs 物理尺寸元数据 | MUST | V3.1 §5.2 明确规定，docx 嵌入硬要求 |
| 单栏图宽度 ≤ 14cm，通栏 ≤ 16cm | MUST | V3.1 §5.2 明确规定 |
| 图注末尾标注数据来源 | MUST | V3.1 §5.2（"来源：图注末尾标注数据来源"） |

#### 数据图表类型选择

| 约束 | 等级 | 理由 |
|------|------|------|
| 饼图 > 5 项时取 Top5+其他 | MUST | V3.1 §5.3 明确规定 |
| 使用 3D 效果的图表 | SHOULD | 数据可视化最佳实践——3D 扭曲数据比例 |
| 双 Y 轴（无明确关联的不同量纲） | SHOULD | 极易误导读者 |
| 选用决策矩阵中标记为"反模式"的图表类型 | SHOULD | 记录偏差，除非有充分理由 |

### 4.3 等级与检查方式的映射

| 等级 | 自动检查 | 人工检查清单 | 阶段 8 红队 |
|------|---------|-------------|------------|
| MUST | 全部自动检查，FAIL 阻塞 | 不在人工清单中出现（已自动化） | 抽样复核自动检查结果 |
| SHOULD | 能做自动化的尽量做（WARN 不阻塞） | 无法自动化的 SHOULD 在此列出 | 逐条确认偏差是否有合理理由 |
| MAY | 不做自动检查 | 作为"建议项"列出 | 不检查 |

                    │  1. 读取跨图颜色映射注册表         │
                    │  2. 逐图对比实际色值               │
                    │  3. 检查字体一致性                 │
                    │  4. 产出图表一致性风险清单         │
                    └──────────────┬──────────────────┘
                                   │
                    ┌──────────────▼──────────────────┐
                    │   阶段 9 定稿（最终验证）          │
                    │                                   │
                    │  1. 运行 check_cross_figure_      │
                    │     consistency.py                │
                    │  2. 确认所有 SHOULD 级偏差有记录   │
                    │  3. 图表合规性最终 PASS            │
                    └──────────────────────────────────┘
```

### 7.2 时序约束

| 环节 | 时序约束 | 原因 |
|------|---------|------|
| 跨图颜色映射注册表创建 | 在阶段 6 **第一张图生成之前** | 必须先建立实体-颜色映射，后出图；出图后才能回填 |
| rcParams 模板加载 | 在每个 `plt.savefig()` **调用之前** | matplotlib 的 rcParams 在 `plt.style.use()` 后立即生效，需在 savefig 前确保已加载 |
| 自动检查脚本运行 | 在每张图 PNG 导出**之后、阶段质量门禁检查之前** | 检查需要 PNG 文件已存在 |
| 跨图一致性检查 | 在阶段 8 红队审查时（**所有图完成后**） | 需要所有图都产出后才能做跨图对比 |

## 9. 扩展性设计

### 9.1 新增配色主题

**扩展方式**：在 `color-registry.yaml` 中追加新的 `theme` 条目，不修改已有条目。

```yaml
# 扩展示例：在灰度色板（默认）之外追加"深色终端"主题
themes:
  grayscale:           # 已有（默认，V3.1 基线）
    background: "#FFFFFF"
    ...
  dark-terminal:       # 新增——不影响现有约束
    background: "#1a1a2e"
    text_primary: "#e0e0e0"
    ...
```

**下游影响**：
1. Layer 2 各工具适配层需追加该主题的映射表
2. Layer 3 检查脚本需追加该主题的允许色值范围
3. 不影响现有 `grayscale` 主题的任何约束

### 9.2 新增图表类型

**扩展方式**：在 `chart-type-decision-matrix.yaml` 中追加新行（数据类型 → 新图表类型）。

```yaml
# 扩展示例：追加"桑基图"作为数据流可视化的新推荐类型
- data_type: "多步骤流量转移"
  recommended: "sankey"
  anti_patterns: ["stacked_bar", "multiple_pie"]
  conditions: "≥ 3 个阶段，流量在阶段之间有分支/合并"
  skeleton_index: "skeleton-sankey.py"
```

**下游影响**：
1. 在 `chart-skeletons/` 中新增对应的代码骨架文件
2. 检查脚本无需修改（通用检查不依赖特定图表类型）
3. 不影响现有决策矩阵条目

### 9.3 新增出图工具

**扩展方式**：在 Layer 2 新增该工具的适配层配置，在 Layer 3 追加或扩展检查脚本。

```text
references/chart-quality/
  <new-tool>-style-adapter.yaml    # Layer 2：将颜色/字体注册表映射为该工具的配置格式
  scripts/check_<new-tool>.py      # Layer 3（可选）：如果新工具有独特的检查需求
```

**不修改的部分**：
- Layer 1 所有注册表（色值/字体/决策矩阵是工具无关的）
- 已有工具的适配层配置
- 已有检查脚本
- SKILL.md 中阶段 6 的工具选择表追加一行

### 9.4 约束等级调整

**扩展方式**：在 `SKILL.md` 质量门槛清单中修改对应检查项的等级标注，或（如果约束等级存储在配置文件中）在对应的 YAML 文件中修改 `severity` 字段。不修改检查脚本逻辑。

### 9.5 扩展性原则

| 原则 | 说明 |
|------|------|
| **注册表增行不改列** | 配色注册表追加主题、决策矩阵追加类型时，只增行不增列（schema 向后兼容） |
| **检查脚本独立编译** | 每个检查脚本是独立可运行的 Python 文件，可以通过阶段质量门禁任意组合调用顺序 |
| **适配层单向依赖** | Layer 2 的适配层只依赖 Layer 1 注册表，不反向依赖，不横向依赖 |
| **人工清单可热更新** | `chart-quality-checklist.md` 是纯 Markdown，追加新检查项不需要修改任何代码 |

## 10. 架构风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 人工检查清单被跳过（用户忽略 SHOULD 级检查） | 高 | 中——图表一致性下降但不阻塞交付 | 阶段 8 红队将图表一致性作为独立维度强制检查，阶段 9 最终检查要求确认所有 SHOULD 偏差有记录 |
| 配色注册表中色值与工具实际渲染色值有偏差（显示器伽马/色彩空间差异） | 中 | 低——轻微色差，印刷时进一步压缩为灰度 | 检查脚本使用宽松色差阈值（Delta-E ≤ 5），不在像素级精确匹配上阻塞 |
| 跨图颜色映射注册表维护成本高（多图多实体时逐条录入繁琐） | 中 | 中——可能被跳过或填错 | 注册表模板预填写常见的架构层名称（"感知层""网络层""平台层""数据层""应用层""安全层"），自动分配色值；AI 在阶段 6 出图时自动回填 |
| drawio Mermaid 主题变量覆盖范围有限（Mermaid 的主题变量不完全覆盖所有样式属性） | 中 | 中——部分样式属性无法通过主题变量控制 | 在 drawio 样式约束规格中明确列出 Mermaid 主题变量的覆盖范围和不可覆盖属性；对不可覆盖属性降级为 SHOULD 或 MAY |
| 图表代码骨架库可能因 matplotlib 版本更新而失效 | 低 | 中——骨架无法运行 | 骨架代码使用稳定的 matplotlib API，避免实验性接口；每个骨架包含版本声明注释 |
| 检查脚本的色值合规检查无法解析 drawio 内部 XML（drawio 可能将颜色编码在不可预测位置） | 中 | 低——检查失败但不阻塞 | `check_color_compliance.py` 对 drawio 产物标记为"人工检查"，不自动判定 FAIL |

## 11. 分阶段实施建议

| 阶段 | 交付物 | 风险 | 依赖 |
|------|--------|------|------|
| **阶段 A：Layer 1 注册表** | 配色注册表、字体注册表、图表类型决策矩阵的初稿 YAML | 低——纯设计文档，无运行时依赖 | V3.1 格式规范 §5.2（已就绪） |
| **阶段 B：matplotlib 约束** | rcParams 模板 + 代码骨架（10 个文件）+ check_png_quality.py | 中——matplotlib rcParams 在不同环境的字体可用性差异 | 阶段 A 完成（需要色值和字体规格） |
| **阶段 C：架构图工具约束** | drawio 样式规格 + fireworks 样式参数 + Mermaid 主题配置 | 中——Mermaid 主题变量覆盖范围限制 | 阶段 A 完成 |
| **阶段 D：自动检查脚本** | check_color_compliance.py + check_cross_figure_consistency.py + 人工检查清单 | 高——检查脚本的正确性直接影响流程阻塞与否 | 阶段 A+B 完成 |
| **阶段 E：SKILL.md 集成** | SKILL.md 中所有侵入点的实际修改（质量门槛追加、步骤追加） | 低——追加式修改，不替换现有内容 | 阶段 A-D 全部完成（确保所有被引用的文件路径存在） |
| **阶段 F：红队+最终检查** | 阶段 8 新增维度 + 阶段 9 新增检查项 | 低——追加式修改 | 阶段 E 完成 |

**推荐实施顺序**：A → B+C（并行）→ D → E → F

## 12. 设计决策记录

| # | 决策 | 选项 | 选择 | 理由 |
|---|------|------|------|------|
| D1 | 约束注册表存储格式 | JSON / YAML / INI | YAML | 人类可读写 + 支持注释 + 已在项目 tool-paths.json 中使用 JSON 但 YAML 对多行字符串和注释更友好 |
| D2 | 配色注册表与工具适配层的关系 | 注册表生成适配层（脚本自动）/ 注册表被适配层引用（AI 读取） | AI 读取引用 | 减少脚本依赖——AI 在出图时直接读取 YAML 并翻译，不需要中间生成步骤 |
| D3 | 检查脚本对 MUST 级失败的处理 | 退出非零码 / 返回 JSON 报告 | 返回 JSON 报告 + 可选 --strict 退出码 | 与 md-to-docx v2 的 exit code 契约一致（报告承载结果，exit code 由顶层编排决定） |
| D4 | 跨图颜色映射注册表的维护者 | 用户手动维护 / AI 自动维护 | AI 自动维护 | 阶段 6 出图时 AI 自动提取实体名并分配色值，用户只需确认 |
| D5 | 代码骨架库与决策矩阵的绑定方式 | 硬编码文件路径 / 索引列名引用 | 决策矩阵中通过 `skeleton_index` 列引用文件名 | 简单、可读、易维护 |
| D6 | 人工检查清单的触发机制 | 阻塞性（不勾选不进下一阶段）/ 记录性（勾选后继续） | 记录性（MUST 级通过自动检查阻塞，SHOULD 级人工勾选不阻塞但记录） | 避免人工清单成为瓶颈 |
| D7 | drawio 约束的实现方式 | 通过 Mermaid 语法注入 / 通过 mxGraphModel XML 注入 / 两者共存 | 两者共存——优先 Mermaid 主题变量（简单图），备选 XML 样式片段模板（复杂图） | Mermaid 覆盖大多数简单流程图，XML 用于需要精确控制的复杂架构图 |
