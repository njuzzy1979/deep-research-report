## 1. CLI 接口设计

### 1.1 命令行签名

```
用法: python -m md2docx <input.md> [output.docx] [选项]
       （等价 shim：scripts/md2docx.py <input.md> [output.docx] [选项]，供不熟悉 -m 用法的调用方）
```

**位置参数**

| 参数 | 类型 | 必填 | 说明 |
|------|------|:---:|------|
| `input` | 路径 | 是 | 输入 Markdown 文件路径 |
| `output` | 路径 | 否 | 输出 docx 路径；缺省 = `input` 同目录、同文件名（扩展名替换为 `.docx`），如 `final-report.md` → `final-report.docx` |

**可选参数表**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `--figures-dir` | 路径 | `""`（空） | 图片搜索**追加候选目录**（不替换主解析路径，主路径始终是"以 md 文件所在目录解析相对路径"，见 02 §A.4） |
| `--config` | 路径 | `""`（空，自动探测，见 §3.3） | 显式指定 YAML 配置文件路径 |
| `--report-path` | 路径 | `""`（空） | 转换报告输出路径；缺省 = `<output 去扩展名>.conversion-report.md` |
| `--dump-intermediate` | flag | `false` | 额外写出清理后的中间 Markdown（调试用），路径固定为 `<input 去扩展名>-cleaned.md`（架构 N2：可选调试 dump，非管道必经） |
| `--strict` | flag | `false` | ERROR 级问题直接升级为 FATAL 中止（不做降级渲染），供定稿前的"零妥协"复核使用 |
| `--allow-missing-figures` | flag | `false` | 图片文件缺失时降级为 WARNING + 文字占位段，而非 `E-IMG-01` 默认中止（02 §A.6 #2） |
| `--caption-field-mode` | 枚举 `text\|field` | `text` | 题注实现档位：`text`=Tier1（默认，推荐）；`field`=Tier2（SEQ 域+书签+REF 交叉引用），见 §2.5 |
| `--appendix-page-break` / `--no-appendix-page-break` | flag 对 | `true` | 每个附录 H2 是否独立起页（架构 N3） |
| `--figure-max-width-cm` | float | `14.0` | 图片嵌入宽度上限；**参数校验范围 [8.0, 16.0]**（V3.0 单栏 14cm / 双栏通栏 16cm，越界视为参数错误，argparse 阶段直接拒绝，不进入管道） |
| `--title` `--subtitle` `--report-type` `--org` `--version` `--date` `--header-short` | str | `""` | 元数据补充值。**仅在 md 头部元数据块对应字段缺失时生效**，见 §1.2 优先级 |
| `-v` / `--verbose` | flag | `false` | 控制台输出提升到 DEBUG 级（含全部 INFO 台账） |
| `-q` / `--quiet` | flag | `false` | 控制台仅在非 0 退出码时输出；报告文件不受影响，始终完整写出 |
| `--version` | flag | — | 打印转换器版本号并退出（argparse 标准动作） |
| `-h` / `--help` | flag | — | 帮助信息（argparse 标准动作） |

`--verbose` 与 `--quiet` 互斥（同时给出报参数错误）。

### 1.2 元数据来源优先级与降级策略

按 design-brief 明确要求的优先级（**md 头部元数据块 > CLI 参数 > 缺省值**，本设计将"缺省值"细化为两级）：

```
字段最终值 = md头部元数据块中该字段（若存在非空值）
           或 CLI 对应参数（若显式传入）
           或 YAML 配置 metadata_defaults 中的对应默认值（见 §3.2）
           或 None（该字段判定为"缺失"）
```

取第一个非空值，**不做"CLI 覆盖 md"的常规 CLI 工具行为**——这是本项目对通用 CLI 惯例的一处刻意偏离：md 头部元数据块是"这份报告真实写了什么"的第一手来源，CLI 参数只用于"md 完全没提供该字段时的补充/兜底"，理由：若 CLI 参数能覆盖 md 已写明的字段，批量转换时一次疏忽的 `--org` 参数会静默篡改一份已经写对的报告封面，且不会有任何 Issue 提示（因为"覆盖"在语义上不是错误）；而现有优先级下，CLI 覆盖 md 需要 md 字段本身为空这一前提，篡改真实数据的路径被结构性封死。

**字段级降级规则**（对照 `data_full.md` §1.3 实测事实）：

| 字段 | md 来源正则（02 §D.3） | 缺失处理 |
|------|----------------------|---------|
| `title`（标题） | H1 全文 | md 无 H1 → **FATAL**（架构 §6.1 已定义："无 H1 主标题"是 FATAL 场景，无法产出有意义封面） |
| `subtitle`（副标题） | `**副标题**：值` | 缺失 → 封面不渲染该行（design-brief §A 明确要求）+ `INFO` |
| `report_type`（报告类型） | `**报告类型**：值` | 缺失 → 用 CLI/YAML 兜底；仍缺失则不渲染该行 + `INFO` |
| `organization`（机构） | `**编制机构**：值` | 同上；YAML `metadata_defaults.organization` 是最常用兜底位（项目机构名较稳定，适合放配置） |
| `version` / `date`（版本/日期） | `**版本**：V1.0 \| 2026年7月` 二次拆分（正则 `^(V[\d.]+)\s*[|｜]\s*(.+)$`） | 拆分失败 → 整值存入 `version`，`date` 为 `None`；两者皆缺失 → 该行不渲染 + `WARNING`（版本号缺失比副标题缺失更值得关注） |
| `header_short`（页眉简称） | 无独立 md 语法，来源仅 CLI/YAML | 缺失 → 从 `title` 自动截断（策略见 §2.4） |

### 1.3 退出码约定

| 退出码 | 含义 | 触发场景 |
|:---:|------|---------|
| `0` | 成功（允许存在 WARNING/INFO） | 全部图表/标题/表格正常处理，无 ERROR |
| `1` | 完成但含 ERROR（非 `--strict`） | 如图片缺失且未开 `--allow-missing-figures`（此时是否会晋升为 2 见下） |
| `2` | **FATAL**——管道内部致命错误（无法产出有意义 docx）**或** CLI 参数本身不合法 | 输入文件不存在、UTF-8 解码失败、无 H1 主标题、`--figure-max-width-cm` 越界、未知参数等 |

**裁决说明（I10）**：CLI 参数级错误（argparse 自身校验失败：未知参数/类型错误/必填缺失/`--figure-max-width-cm` 越界）沿用 argparse 默认行为，其退出码本就是 `2`（Unix 惯例）；管道内部 FATAL（架构 §6.1）也定义为 `2`。二者语义高度接近（"因为输入/参数有问题，未能正常产出"），**不强行拆分出独立的第 3 个退出码**，仅在错误信息来源上区分：参数错误走 argparse 的 stderr 直接提示，管道 FATAL 则连同已收集的 Issue 一起写入 `conversion-report.md`（该文件在 FATAL 时依然产出，记录失败前的诊断信息）。

`--strict` 模式下，任何本应为 `ERROR` 级的处理直接短路为 `FATAL`（退出码从 1 提升为 2），即"零妥协"模式不接受任何降级渲染。

### 1.4 控制台输出编码安全策略

对应 `tech_full.md` 第13点实测风险（Windows cp936 控制台下裸 print 中文/emoji 可能 `UnicodeEncodeError`）：

```python
# 唯一入口：iotools.console_out(msg: str) —— cli.py 启动时的一次性尝试 + 逐条兜底
try:
    sys.stdout.reconfigure(encoding='utf-8', errors='replace')
    sys.stderr.reconfigure(encoding='utf-8', errors='replace')
except (AttributeError, ValueError):
    pass  # 极旧终端无 reconfigure；下方逐条兜底仍然生效

def console_out(msg: str, stream=sys.stdout):
    stream.write(msg)          # reconfigure 成功时直接安全
    # 若前置 reconfigure 失败（理论场景），退化为：
    # stream.buffer.write(msg.encode(stream.encoding or 'utf-8', errors='replace'))
```

- 策略：宁可让个别生僻字/emoji 在传统 cmd.exe（cp936）下显示为 `?` 或被 `replace`，**绝不允许因控制台输出而中断已完成的转换**——docx 文件本身早已正确落盘（tech#13 结论：docx 内部 UTF-8 天然安全，风险只在打印环节）。
- 全项目唯一允许 `print`/`sys.stdout.write` 的模块是 `cli.py`（经 `console_out` 包装）；`report.py` 写报告文件走 `iotools.write_text_binary`（`wb` + `encode('utf-8')`），不经过控制台编码路径。

### 1.5 CLI 使用示例

```bash
# 最简单用法：零参数，全部走优先级降级
python -m md2docx research/drafts/final-report.md

# 显式指定输出路径 + 严格模式（定稿前复核）
python -m md2docx research/drafts/final-report.md output/final-report.docx --strict

# 图片目录被移动过，追加候选路径
python -m md2docx research/drafts/final-report.md --figures-dir D:/backup/figures

# md 未提供机构名时的兜底（仅当 md 头部元数据块无该字段才生效）
python -m md2docx report.md --org "遨天科技" --report-type "深度研究报告"

# 开启 Tier2 增强题注（SEQ域+REF交叉引用），关闭附录独立分页
python -m md2docx report.md --caption-field-mode field --no-appendix-page-break

# 调试：保留中间清理产物 + 详细日志
python -m md2docx report.md --dump-intermediate -v
```

**首行缩进字符模式双写实现**（tech#1 关键坑）：

```python
# 长度值：按 11pt 正文两个汉字宽度估算（1 字符 ≈ 1 em ≈ 当前字号），
# 2 字符 ≈ 2 × 11pt = 22pt ≈ 0.777cm，四舍五入取 Cm(0.74)（与 tech_full.md 建议值一致）
paragraph_format.first_line_indent = Cm(0.74)   # 长度模式，兼容性兜底
ind = paragraph_format._element.get_or_add_pPr().get_or_add_ind()
ind.set(qn('w:firstLineChars'), '200')          # 字符模式，Word 优先读取此值
```

**决策说明（正文对齐，未上报，风险低）**：V3.0 未明确规定正文段落的左右对齐方式，仅规定标题"顶格"（左对齐，与调研报告 GB/T 7713.1-2025 §6.5.2"顶格排"一致）。本设计对 Body Text 采用**两端对齐**（`WD_ALIGN_PARAGRAPH.JUSTIFY`），是中文学术报告的通行惯例性默认，非 V3.0 强制条款，`config.py` 提供 `body_alignment: justify|left` 开关供未来调整。

**决策说明（Quote 边框颜色，I7，低优先级已上报）**：V3.0 §3.3 只写"灰色边框左侧线"未给色值，`#BFBFBF` 是本设计的推断值（Word 默认灰色边框常见色阶），非规范硬性数字。

### 2.3 封面规格

**顶部留白 6cm 实现方式（I11 裁决）**：

```
封面第 1 个元素 = 一个空段落，段落属性 space_before = Cm(6.0)，space_after = 0，
                 字号任意（不影响留白，取 Normal 默认即可），无文字内容
其后紧跟标题段落（无需再叠加任何 space_before）
```

采用"单个空段落 + `space_before`"而非"敲若干个回车"（旧 `pitfalls.md` 问题12 修复用的"顶部留白6行"做法），理由：多个空段落的累积高度依赖 Word 默认行距换算，存在 `Cm`/`Pt` 舍入误差累积风险（tech#16 issue #687），且"6 个空段落"这一数字本身与"6cm"没有确定性的数学对应关系（取决于所用样式的行高）；`space_before=Cm(6.0)` 是唯一到毫米级精确、无累积误差的写法。

**封面五要素**（全部来自 `MetadataIR`，§1.2 优先级降级后的最终值；任一字段为 `None` 则跳过该行，不留空段落占位）：

| 要素 | 字号 | 字重 | 颜色 | 对齐 | 段前/段后 |
|------|------|------|------|------|----------|
| 报告标题 | 28pt | Bold | #000000 | 居中 | 0pt / 12pt |
| 英文副标题 | 14pt | 常规 | #000000 | 居中 | 0pt / 24pt |
| 报告类型 | 16pt | 常规 | #000000 | 居中 | 0pt / 36pt |
| 机构名 | 14pt | Bold | #000000 | 居中 | 0pt / 6pt |
| 版本号+日期 | 11pt | 常规 | #000000 | 居中 | 0pt / 0pt |

版本号+日期一行的拼装格式：`{version} | {date}`（若 `date` 为 `None`，仅渲染 `{version}`；若 `version` 也为 `None`，整行跳过）。

**封面所在节技术方案**：封面独占 `SectionSpec(kind=COVER)`，是文档的第 1 个（默认）section。渲染时**不调用** `section.header` / `section.footer` 属性（tech_full.md 第4点推荐路径："保持引用不存在的状态"），不设置 `pgNumType`（无页脚故无需页码格式）。封面结束后的换页由**下一节的分节符**触发（`doc.add_section(WD_SECTION.NEW_PAGE)`），而非 `PageBreakIR`——`cover.py` 自身不调用任何分页 API（与 01 号文档 §5.1 "cover.py 不调用任何分页 API"职责卡一致）。

**无密级字段**：封面渲染函数的输入 `MetadataIR` schema 中本就不存在密级字段（架构 §4.3 已确认），物理上没有可渲染的密级数据。

### 2.4 分节与页码方案

**裁决（I1）：4 节而非架构默认草案的 3 节**

```
Section 1: COVER   —— 无页眉、无页脚、无页码格式
Section 2: ABSTRACT —— 摘要正文；有页眉（简称+底线）；页脚罗马数字 start=1
Section 3: TOC      —— 目录 + 图表目录（若生成）；无页眉；页脚罗马数字续接（不设 start，延续上一节）
Section 4: BODY     —— 正文各章 + 附录；有页眉；页脚阿拉伯数字 start=1
```

拆分原因：V3.0 §6.1 明确"封面和目录页无页眉"，但摘要页应有页眉（条款只排除了"封面和目录页"，未排除摘要页）；若摘要与目录同属一节，页眉要么两页都有要么两页都无，无法满足条款字面要求，故必须在摘要与目录之间插入一个新的分节符。01 号文档 §10 已预留"若页眉页脚设计文档要求目录页无页眉而摘要有，只需在此加一节，渲染器零改动"的接口，本次拆分是对该预留接口的直接回填，**不需要修改渲染器代码路径，只需要 `assemble/breaks.py` 的 `SectionPlan` 多产出一个 `SectionSpec`**。

**各节 `is_linked_to_previous` 精确设置**：

| 节 | header.is_linked_to_previous | footer.is_linked_to_previous | pgNumType | 页眉内容 | 页脚内容 |
|---|---|---|---|---|---|
| COVER | 不触碰（保持未定义） | 不触碰（保持未定义） | 不设置 | 无 | 无 |
| ABSTRACT | `False`（新建独立定义） | `False`（新建独立定义） | `fmt=lowerRoman, start=1` | 简称右对齐+1pt黑色底线 | PAGE 域，居中 |
| TOC | `False`（新建**空**定义——不能用 `True` 继承 ABSTRACT 的简称内容，必须显式独立且留空） | `False`（新建独立定义，仍需页码——见下方说明） | `fmt=lowerRoman`（**不设 `start`**，延续摘要节页码序列） | 无（空页眉，无文字无边框） | PAGE 域，居中（罗马数字续接） |
| BODY | `False`（新建独立定义） | `False`（新建独立定义） | `fmt=decimal, start=1` | 简称右对齐+1pt黑色底线 | PAGE 域，居中 |

**决策说明（TOC 节页脚是否显示页码）**：V3.0 §一 只写"摘要……独立编页（罗马数字页码）"，未明确目录页脚是否显示页码。本设计判定目录与摘要同属"前置部分"，**页脚页码延续显示**（只有页眉被排除），理由：若目录页完全无页码，读者翻阅文档会遇到"页码序列出现空洞"的体验问题，且 V3.0 §10.3 检查清单"页码正确（摘要罗马数字，正文阿拉伯数字）"暗示前置部分应有连续罗马页码。

**页眉简称机制（I2 裁决：固定简称，非 STYLEREF）**：

```
header_short 来源 = MetadataIR.title_short（§1.2 已完成优先级降级）
若仍为 None → 自动截断策略：
    取 title 前 16 个字符（中文按字符数不按词边界，因中文无显式词边界）
    若发生截断（原标题长度 > 16）→ 追加 "…"
```

**不采用 STYLEREF 动态获取当前章标题**（tech_full.md 提到的可选简单字段机制），理由：
1. V3.0 条款字面是"报告简称"（文档级固定值），不是"当前章节标题"（运行标题/running head 的语义），二者是不同概念；
2. 本设计的标题渲染文本含编号前缀（"第三章 XXX"），STYLEREF 取得的内容会带编号，不是"简称"效果；
3. 固定简称只需一次性写入所有含页眉的节，实现与校验成本远低于 STYLEREF 依赖运行时按 Heading 1 内容动态计算的方案，且不引入"页面跨章节边界时取值歧义"的边界情况。

页眉底线：`w:pBdr/w:bottom`，`sz=8`（1pt）、`color=000000`、`space=1`（tech#5 已验证）。

### 2.5 目录规格

**TOC 域精确 instrText**：

```
TOC \o "1-3" \h \z \u
```

- `\o "1-3"`：仅收录大纲级别 1-3，对应 Word 样式 Heading 1/2/3（=md 语义章/节/小节），Heading 4（段落小标题）与 Heading 5 不入目录——与 V3.0 §3.2"H4 仅段首强调，不入目录"完全对应。
- `\h`：生成可点击超链接（V3.0"可跳转"要求）。
- `\z`：隐藏 Web 版式下的制表符前导符（标准做法，不影响打印/阅读视图）。
- `\u`：无法从 `\o` 读出大纲级别时退化使用样式中的大纲级别，兜底项。

**四态结构（5 个 run）**：

```xml
<w:r><w:fldChar w:fldCharType="begin"/></w:r>
<w:r><w:instrText xml:space="preserve">TOC \o "1-3" \h \z \u</w:instrText></w:r>
<w:r><w:fldChar w:fldCharType="separate"/></w:r>
<w:r><w:t>（目录将在打开文档并更新域后自动生成；可右键此区域选择"更新域"，或全选正文后按 F9）</w:t></w:r>
<w:r><w:fldChar w:fldCharType="end"/></w:r>
```

`separate` 后的占位结果**不留空**（tech 结论：python-docx 无法计算 TOC 结果），而是写入一段中文操作提示——避免用户打开文档看到空白或 Word 原生"错误！未找到目录项。"提示语造成困惑。同时全局 `settings.xml` 写入 `w:updateFields=true`（提示 Word 打开时更新域），并在转换报告的"待人工确认项"中列出"首次打开后按 F9 更新目录/页码域"（WPS 对该开关遵循度未定，tech#14，报告需明确提示不能假设 WPS 静默生效）。

**图表目录（I3 裁决：文本条目 + PAGEREF 页码域混合方案）**

- **是否生成**：样本 9 图 + 3 张有题注的正文表 = 12，超过 V3.0 §一"如有 10 个以上图表才生成"的阈值 → **判定应生成**。
- **技术矛盾与折中**：真正的"Word 原生图表目录"（`TOC \c "Figure"` / `TOC \c "Table"`）依赖题注文字内嵌 `SEQ` 域标识符供 `\c` 扫描——这要求 Tier2（`--caption-field-mode=field`）。默认 Tier1（文本题注）下没有 `SEQ` 标记，`TOC \c` 域技术上无法产出正确结果。
- **折中方案**：不因此彻底放弃图表目录，也不用"手写静态列表"这种明确违反 V3.0 §10.1"禁止手动输入目录"的做法，而是采用一种介于两者之间的方案——
  1. 每个 `FigureIR`/`TableIR`（无论 Tier1 还是 Tier2）**在渲染时都打一个书签**（`bookmark_name` 字段架构已预留，本设计明确其在 Tier1 下也被消费，用途仅限图表目录，不用于正文交叉引用）；
  2. 图表目录页的每一条目 = **静态文本**（题注全文，取自 IR，非硬编码）+ 制表符 + **`PAGEREF {bookmark_name} \h` 域**（真正 Word 域，取得该图表实际所在页码，翻页增删内容后按 F9 更新即可保持正确）；
  3. 条目文本本身不是 Word 域，但页码部分是——这不是"完全域自动生成条目"，故在 V3.0 §10.1 对照表中标注为**部分覆盖/有意变通**（见 §4），但避免了"页码写死后来源失真"这一最容易过时、最不可接受的手工痕迹。
- **结构**：单一"图表目录"页（TOC 节内，罗马页码续接），内部两个不入正文 TOC 的子标题——"图目录"与"表目录"（渲染为加粗小标题，非 Heading 样式，不占用 Heading1-3 编号体系）：

```xml
<w:r><w:fldChar w:fldCharType="begin"/></w:r>
<w:r><w:instrText xml:space="preserve"> PAGEREF fig_1_1 \h </w:instrText></w:r>
<w:r><w:fldChar w:fldCharType="separate"/></w:r>
<w:r><w:t>1</w:t></w:r>   <!-- 占位页码，Word 更新域后回填真实值 -->
<w:r><w:fldChar w:fldCharType="end"/></w:r>
```

- **开关**：`--generate-figures-table-toc {auto,always,never}`（默认 `auto`：数量≥10 才生成；`always`/`never` 供强制控制），YAML 对应 `behavior.generate_figures_table_toc`。

### 2.6 表格规格

**宽度与列宽**：

```
可用页宽 = section.page_width - left_margin - right_margin = 21.0cm - 3.17cm - 2.54cm = 15.29cm
表格目标宽度 = 15.29cm × 0.90 = 13.761cm（tblPr/w:tblW，w:type="dxa"，twips = 13.761 × 566.929 ≈ 7802）
列宽策略：等宽平分（表格目标宽度 / n_cols），cell.width 与 tblGrid/gridCol/@w:w 必须同步设置
         （tech#6 提醒：仅设 cell.width 而不同步 gridCol，部分渲染器下可能不生效）
```

未对"按内容长度自适应列宽"做特殊设计（YAGNI）：样本 6 张表列数 2-5 列，等宽平分已足够，避免过度设计。

**表头行**：微软雅黑 10pt Bold #000000，白底（不写 `shd` 元素，默认白底），单元格垂直居中 + 文字居中。

**数据行**：宋体 10.5pt #000000，交替底纹——**裁决**：第 1 条数据行（表头正下方）为白底，第 2 条为 `#F2F2F2`，往后交替（数据行索引从 0 起，偶数=白，奇数=灰）。V3.0 未规定起始色，取此为默认惯例。

**边框（全框线，逐项 sz 值）**：

| 边线 | 粗细 | `w:sz` | 颜色 | 实现层级 |
|------|-----|--------|------|---------|
| 顶线 | 1.5pt | `12` | `000000` | `tblPr/w:tblBorders/w:top` |
| 底线 | 1.5pt | `12` | `000000` | `tblPr/w:tblBorders/w:bottom` |
| **表头下分隔线** | 1pt | `8` | `000000` | **表头行 `tcPr/w:tcBorders/w:bottom`**（单元格级，逐单元格设置，覆盖表级默认；见下方 I4 说明） |
| 内部横线（非表头下） | 0.5pt | `4` | `000000` | `tblPr/w:tblBorders/w:insideH` |
| 内部竖线 | 0.5pt | `4` | `000000` | `tblPr/w:tblBorders/w:insideV` |
| 左/右外框 | 0.5pt | `4`（推断值，V3.0 未单独规定左右外框粗细，与内部竖线取同粗细） | `000000` | `tblPr/w:tblBorders/w:left`/`w:right` |

**裁决 I4（OXML 实现关键点）**：V3.0 要求"表头下分隔线 1pt"区别于"内部横线 0.5pt"，但 OOXML 的 `tblBorders/insideH` 是**表格级统一设置**，无法只对"第一条内部横线"单独取值。解决方案：`tblBorders.insideH` 按 0.5pt 设为**默认值**（适用于除表头下以外的所有内部横线），**表头行的 `tcPr/w:tcBorders/w:bottom` 单独显式设为 1pt**——OOXML 中单元格级边框优先级高于表格级，可实现"表头下这一条线比其余内部横线粗"的差异化效果，无需特殊 hack。

**对齐（I5，需编排器确认）**：V3.0 §5.1"对齐"行字面写"文字居中'列居中+行居中'"，本设计**遵循字面规定，全表格文字（含首列）居中**，不做"首列左对齐"的例外——因为规范条款本身没有给出例外空间，若转换器擅自引入例外属于对规范的越权变通，需要明确的用户授权。同时提供 `table_first_col_left_align: bool = False` 配置开关（不改变默认行为，供确认后按需切换）；学术出版惯例中纯文字描述列常见左对齐更易读，这是本设计认为**值得向编排器/用户确认**的一点，非本设计单方面拍板。

**缩进**：Table Header/Table Body 样式不继承 Body Text 的首行缩进（`first_line_indent=0`, `left_indent=0`）。

**跨页表头**：首行 `trPr/w:tblHeader` 设 `w:val="true"`（tech#6 已验证）。

### 2.7 图片规格

| 参数 | 值 | 出处 | 实现路径 |
|------|-----|------|---------|

**图注含数据来源时的排版**：实测样本（`data_full.md` §1.4）9 处图片引用**没有一处**紧跟独立"数据来源："行——来源信息若存在，融合在题注文字本身或后续正文段落中，`FigureIR` schema 因此不设独立 `source_line` 字段（与 `TableIR` 不同）。若来源信息已包含在 alt 文本内，随题注整体渲染；转换器不额外拼装"数据来源："前缀。

**裁决说明（图注颜色 I6）**：design-brief 提出的"V3.0 §十一辅助文字#555555 vs §3.2黑色"张力，经核查 §3.2 是标题层级表（H1-H5），**并不包含图注/表注这一行**，二者不构成真实条款冲突；给出图注/表注颜色的唯一条款是 §十一"辅助文字 #555555｜用途：图注、表注、脚注"。本设计据此裁定 `#555555`，并认为这不是需要"二选一"的冲突，而是 §3.3 表格本身遗漏颜色列、由 §十一补全的关系。仍在 §0 决策清单中列出以供确认，因为该判断本身依赖对条款体系的解读。

### 2.8 题注与交叉引用域方案（D14 分级，最终定案）

| | Tier1（默认，`--caption-field-mode=text`） | Tier2（可选，`--caption-field-mode=field`） |
|---|---|---|
| 题注实现 | 纯文本写入段落 run（"图1-1 xxx"），无 SEQ 域 | `SEQ Figure \* ARABIC` 域包一层书签 |
| 正文交叉引用 | 原文本保留（"（图1-1）"仍是纯文字） | 正文引用模式替换为 `REF fig_1_1 \h`（可点击域） |
| 图表目录支持 | ✓（§2.5 混合方案，书签+PAGEREF，独立于本档位） | ✓（可用真正 `TOC \c` 域） |
| 工作量 | 低 | 高 |
| 与标题编号体系的一致性 | ✓ 一致（标题编号也是文本写入，D3） | ✗ 形成"标题文本编号 + 图表域编号"半文本半域的混合体系 |
| WPS 兼容风险 | 无（无域可言） | 中（域自动更新遵循度不确定，tech#14） |
| 是否改写正文内容 | 否 | **是**——需要识别正文中"（图X-Y）"等散落引用模式并替换为 REF 域，这一操作违反架构 P1"渲染层不改写文本内容"的原则，需要专门新增一个正文预处理子阶段 |

**推荐档位：Tier1（默认）**。理由已在架构 01 号文档 §8.2 给出（体系一致性、工程成本收益比、WPS 风险），本设计在此基础上补充一条接口层理由：Tier2 若要生效，必须先解析正文段落中所有"图表提及"的具体位置与原文片段（即 §E 交叉引用校验算法已经在做的 `XRefMention` 登记），再逐一替换为域——这一步骤会把"只读校验"（02 §E，只产生 Issue 不改内容）升级为"主动改写"，需要新增一条数据流通路，与既有的"阶段4只读校验"契约（01 §2.0）相冲突，成本明显高于表面看到的"多包一层 XML"。

TOC 域、PAGE 域**始终必做**（D14 硬性要求），与 Tier 选择无关。

### 2.9 特殊元素

### 2.10 正文行内格式

| 语法 | 渲染 | 实现路径 |
|------|------|---------|
| `**加粗**` | `run.bold = True` | 原生 API |

### 2.11 "主张—证据—推理"段落

沿用架构 01 号文档 N4 裁定，本节给出接口层的最终渲染细节：

- **默认渲染**：按 Body Text 普通段落渲染，其中"**主张**："/"**证据**："/"**推理**："等加粗片段通过标准行内加粗解析（`bold=True`）自然保留，不额外套用任何框架/底纹/边框。
- **可选增强**（默认关闭）：`claim_evidence_box: bool = False` 配置开关。若打开，`render/special.py` 通过**文本内容匹配**（检测段落是否以"主张—证据—推理"或"主张："等固定短语开头）触发定义框样式渲染。**明确标注**：这是本设计中唯一一处"按内容而非语法结构触发渲染"的例外，与整体"格式与内容解耦"原则（D1 精神）存在张力，之所以允许，是因为样本 0 处使用结构化 Markdown 标记（`data_full.md` §1.7 已确认"主张—证据—推理"纯粹是加粗文字+句号分隔），若坚持"零内容匹配"就只能彻底放弃这一可视化增强；默认关闭确保该例外不会在未经确认的情况下生效。

### 2.12 附录规格

**附录标题渲染格式（I8 裁决）**：`附录A：事实核验台账摘要`（全角冒号），与 `02-algorithms.md` §C.3 已定的渲染格式保持一致。design-brief 任务描述中出现的"附录A 事实核验台账摘要"（空格式）判定为行文示例，不作为裁决依据——跨设计文档一致性优先。

**附录表格**：`TableKind.APPENDIX` 类型，无题注段落、无来源段落，仅渲染表格本体；表格自身的字体/字号/边框/底纹/居中/跨页表头等样式与正文表**完全一致**（V3.0 未对附录表格另立样式标准）。

**附录分页**：默认每个附录 H2 独立起页（架构 N3，`--appendix-page-break` 可关）。

## 3. 配置文件 schema 设计

### 3.1 裁决：是否需要配置文件（YAML vs 纯 CLI）

**裁决：保留 YAML 配置文件**（延续项目既有 `report.yaml` 先例），但**严格切分两类内容**：

| 类别 | 是否可配置 | 存放位置 | 理由 |
|------|:---:|---------|------|
| **V3.0 格式规范常量**（字体名/字号/颜色值/边距/行距/表格边框粗细/域 instrText 等） | **否** | `style_constants.py`（代码常量，无 CLI/YAML 入口） | design-brief 明确提出的裁决点：若这些数值可被 YAML 随意修改，"转换器输出必然符合 V3.0"这一核心保证即失效——用户一次误改的 YAML 会让产出文档"看起来是本工具生成的"但实际不合规，且转换器自身无法察觉（这类改动不是 bug，是"按用户配置正确执行"），信任链断裂 |
| **行为开关**（`strict`/`allow_missing_figures`/`appendix_page_break`/`caption_field_mode`/`figure_max_width_cm` 等） | **是** | YAML `behavior` 块 + CLI 同名参数 | 这些不改变"最终渲染出的具体格式数值是否合规"，只改变处理流程/容错策略/档位选择，安全暴露 |
| **元数据兜底值**（`organization`/`report_type_default`/`header_short` 等） | **是** | YAML `metadata_defaults` 块 + CLI 同名参数 | 项目机构名等信息跨报告复用度高，适合配置化；且其优先级低于 md 头部元数据块（§1.2），不会覆盖真实数据 |

`figure_max_width_cm` 虽然看似"格式参数"，但因其本质是一个**上限约束**（非精确值，图片按纵横比自适应缩放），且已设定 [8,16]cm 的合规校验范围（覆盖 V3.0 "单栏≤14cm/双栏通栏≤16cm"两档），故归入"行为开关"而非"格式规范常量"——用户能调的范围始终落在合规区间内，不存在"改出不合规文档"的风险。

### 3.2 YAML Schema

```yaml
# report.yaml —— 项目级默认配置（可选文件；若不存在则全部使用包内置默认值）
# 优先级（仅对本文件覆盖的字段生效）：CLI 参数 > 本文件 > 包内置 default_config.yaml
# 注意：元数据类字段的最终优先级见正文 §1.2（md 头部元数据块 > CLI > 本文件 > 空）

metadata_defaults:
  organization: "遨天科技"          # str，机构名兜底
  report_type_default: "深度研究报告"  # str，报告类型兜底
  header_short: ""                 # str，页眉简称；空则从标题自动截断（§2.4）

behavior:
  strict: false                          # bool
  allow_missing_figures: false            # bool
  appendix_page_break: true               # bool
  caption_field_mode: "text"               # enum: text | field
  dump_intermediate: false                # bool
  figures_dir: ""                         # str，追加图片搜索候选目录，空则仅用 md 同目录解析
  figure_max_width_cm: 14.0               # float，校验范围 [8.0, 16.0]
  table_first_col_left_align: false        # bool，见 §2.6 (I5)
  generate_figures_table_toc: "auto"        # enum: auto | always | never，见 §2.5
  claim_evidence_box: false                # bool，见 §2.11

report:
  path: ""      # str，空则用 <output去扩展名>.conversion-report.md
```

### 3.3 配置文件查找顺序

```
1. --config <path> 显式指定（最高优先级，找不到文件即报参数错误）
2. 与输入 md 同目录的 report.yaml（若存在）
3. 包内置 default_config.yaml（兜底，随包分发，保证零配置也能运行）
```

字段缺失时逐字段回落到内置默认值（不要求 YAML 文件必须包含全部字段）。
