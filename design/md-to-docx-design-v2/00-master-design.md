## 1. 设计概述

### 1.1 目标
为 deep-research-report skill 阶段 9 重建 Markdown→Word 转换器：输入符合 skill 约定的研究报告 md（如 `research/drafts/final-report.md`）与图表目录，输出完全符合《研究报告格式规范 V3.0》的 .docx + 一份转换报告。旧实现（`markdown_to_docx.py` + `md2docx/` 包）因 FIGURE_MAP 硬编码缺陷（换报告后 8 图静默丢失）已被删除，本设计为全新重写方案。

### 1.2 范围
- Markdown 受控子集解析（ATX 标题/段落/管道表格/图片/`---`/一级列表/加粗斜体行内代码）
- V3.0 全部排版要求（页面/样式/编号/图表/页眉页脚/封面/特殊元素/配色/Word 域技术）
- 三道校验门禁与转换报告；13+1 历史缺陷的结构性规避；反硬编码机制

### 1.3 约束
- Python + python-docx 1.2.0，零第三方解析依赖，零 subprocess，纯本地
- Windows 中文环境（cp936）：全流程二进制 I/O + 显式 UTF-8（硬约束 D10）
- 15 项功能需 OXML 底层手写（tech 实测清单，见 01 §8.2 / tech_full 汇总表）
- SVG 不可嵌入（实测）→ 统一 PNG；脚注不实现（V3.0 ①②③方案已规避）

### 1.4 核心设计思想（五支柱，01 §1.2）
1. 文本域与渲染域物理隔离（根治历史问题 8）
2. 数据驱动 IR、代码零内容硬编码（根治问题 9 与 FIGURE_MAP）
3. 单一触发点：分页 PageBreakIR 唯一生成/消费点、域代码唯一构造函数、I/O 唯一进出口
4. 图/表两条独立题注路径互不共享状态
5. 全程 Issue 留痕，任何自动处置不允许静默

## 2. 数据管道总览（细节见 01）

```
原始md(bytes) ─rb→ 阶段0规范化(BOM/CRLF→LF/utf-8) → 阶段1清理(规则表R-01~R-11)
  → 阶段2解析(Token流) → 阶段3 IR构建(标题分类/编号重编/图表三元组/分页分节规划)
  → 阶段4渲染前校验(只读) → 阶段5渲染(python-docx, 四节:封面/摘要/目录/正文)
  → 阶段6后校验+转换报告(exit 0/1/2)
```

图片三元组 `(图号, 题注, 路径)` 100% 从 `![图X-Y 标题](路径)` 解析（02 §A）；表题注/来源行从 `**表X-Y 标题**`/`*数据来源：…*` 邻接块关联（02 §B）；标题兼容式剥离手动编号后统一重编（02 §C）。
